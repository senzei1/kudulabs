Create the Feature Store Table

This table serves as both the feature store (behavioral attributes) and the scoring output (churn_score, risk tier). A single Kudu table replaces what would traditionally be a nightly Hive batch + a separate serving layer.

Impala SQL 
CREATE TABLE subscriber_features (
  subscriber_id     STRING,
  -- Demographic / account info
  region            STRING,
  plan_type         STRING,
  account_age_days  INT,
  -- Behavioral features (continuously updated)
  avg_monthly_revenue    DECIMAL(10,2),
  revenue_trend          STRING     COMMENT 'increasing | stable | declining',
  total_calls_30d        INT        DEFAULT 0,
  total_data_mb_30d      DOUBLE     DEFAULT 0,
  complaint_count_30d    INT        DEFAULT 0,
  last_complaint_date    TIMESTAMP,
  call_drop_rate_7d      FLOAT      DEFAULT 0,
  days_since_recharge    INT,
  competitive_exposure   BOOLEAN    DEFAULT FALSE,
  nps_score              INT,
  last_nps_date          TIMESTAMP,
  -- Model output (written by Spark ML)
  churn_score            FLOAT,
  churn_risk_tier        STRING     COMMENT 'high | medium | low',
  last_scored_at         TIMESTAMP,
  -- Retention action tracking
  retention_contacted    BOOLEAN    DEFAULT FALSE,
  retention_offer        STRING,
  retention_outcome      STRING     COMMENT 'accepted | declined | pending | null',
  last_contacted_at      TIMESTAMP,
  PRIMARY KEY (subscriber_id)
)
PARTITION BY HASH (subscriber_id) PARTITIONS 16
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = 'KUDU_MASTER:7051',
  'kudu.num_tablet_replicas' = '3'
);

Design note: No range partitioning here because this is a current-state table, not a time-series. Each subscriber has exactly one row that gets updated as new signals arrive. Hash partitioning alone distributes the workload evenly.

Seed the Feature Store with Historical Data

In production, the feature store would be populated incrementally. For this lab, we seed it with synthetic subscriber data using PySpark.

PySpark — seed_features.py 
from pyspark.sql import SparkSession
from pyspark.sql.functions import *
from pyspark.sql.types import *
import random

spark = SparkSession.builder \
    .appName("Seed Churn Feature Store") \
    .getOrCreate()

# Generate 50,000 synthetic subscribers
num_subscribers = 50000
regions = ['San Jose', 'Heredia', 'Alajuela', 'Cartago', 'Guanacaste',
           'Puntarenas', 'Limon', 'Bogota', 'Medellin', 'Lima']
plans = ['Prepaid Basic', 'Prepaid Plus', 'Postpaid Standard',
         'Postpaid Pro', 'Enterprise']
trends = ['increasing', 'stable', 'declining']

data = []
for i in range(num_subscribers):
    sub_id = f"SUB-{i:05d}"
    region = random.choice(regions)
    plan = random.choice(plans)
    age = random.randint(30, 2000)
    revenue = round(random.uniform(5.0, 150.0), 2)
    trend = random.choice(trends)
    calls = random.randint(0, 500)
    data_mb = round(random.uniform(0, 50000), 1)
    complaints = random.randint(0, 8)
    drop_rate = round(random.uniform(0, 0.15), 4)
    recharge_days = random.randint(0, 90)
    nps = random.choice([None, random.randint(1, 10)])

    data.append((sub_id, region, plan, age, revenue, trend,
                 calls, data_mb, complaints, None, drop_rate,
                 recharge_days, False, nps, None,
                 None, None, None,
                 False, None, None, None))

schema = StructType([
    StructField("subscriber_id", StringType()),
    StructField("region", StringType()),
    StructField("plan_type", StringType()),
    StructField("account_age_days", IntegerType()),
    StructField("avg_monthly_revenue", DecimalType(10,2)),
    StructField("revenue_trend", StringType()),
    StructField("total_calls_30d", IntegerType()),
    StructField("total_data_mb_30d", DoubleType()),
    StructField("complaint_count_30d", IntegerType()),
    StructField("last_complaint_date", TimestampType()),
    StructField("call_drop_rate_7d", FloatType()),
    StructField("days_since_recharge", IntegerType()),
    StructField("competitive_exposure", BooleanType()),
    StructField("nps_score", IntegerType()),
    StructField("last_nps_date", TimestampType()),
    StructField("churn_score", FloatType()),
    StructField("churn_risk_tier", StringType()),
    StructField("last_scored_at", TimestampType()),
    StructField("retention_contacted", BooleanType()),
    StructField("retention_offer", StringType()),
    StructField("retention_outcome", StringType()),
    StructField("last_contacted_at", TimestampType()),
])

df = spark.createDataFrame(data, schema)

df.write.format("kudu") \
    .option("kudu.master", "KUDU_MASTER:7051") \
    .option("kudu.table", "impala::default.subscriber_features") \
    .option("kudu.operation", "upsert") \
    .mode("append") \
    .save()

print(f"Seeded {num_subscribers} subscribers")
Shell 
spark-submit \
  --packages org.apache.kudu:kudu-spark3_2.12:1.18.1 \
  seed_features.py

Build the NiFi Event Ingestion Flow

Simulate behavioral events arriving from multiple source systems (CRM, billing, network monitoring). Each event type updates different columns in the feature store.

3a. Event Generator

Use a GenerateFlowFile processor to create events. In production, these would come from Kafka topics fed by the CRM, billing system, and network monitoring tools.

Event JSON (complaint event example) 
{
  "event_type": "complaint",
  "subscriber_id": "SUB-${random():mod(50000):format('%05d')}",
  "timestamp": "${now():format('yyyy-MM-dd HH:mm:ss')}",
  "complaint_category": "billing_dispute"
}

3b. RouteOnAttribute — Event Type Router

Route events to different UpdateRecord processors based on event_type. Each processor maps the event payload to the correct feature store columns.

RouteConditionFields Updated
complaint ${event_type:equals('complaint')} complaint_count_30d (increment), last_complaint_date
recharge ${event_type:equals('recharge')} days_since_recharge (reset to 0), avg_monthly_revenue
network_quality ${event_type:equals('network')} call_drop_rate_7d
nps_response ${event_type:equals('nps')} nps_score, last_nps_date

3c. PutKudu — UPSERT to Feature Store

Each route connects to a PutKudu processor configured with UPSERT operation. The UPSERT only updates the columns present in the FlowFile record — other columns retain their existing values.

Partial row updates: Kudu's UPSERT replaces the entire row — it doesn't merge individual columns. To update only specific columns, you need to either: (a) read the current row first and merge, or (b) restructure as separate narrow tables (one per signal source). For this lab, we use approach (a) via an LookupRecord processor with a Kudu lookup service, or handle the merge in a lightweight Spark streaming job.

✓ Checkpoint

  • Events flowing through NiFi at steady rate
  • Feature store rows being updated (check last_complaint_date on random subscribers)
  • No errors in PutKudu failure relationship

Build the Churn Scoring Model (Spark ML)

This step trains a simple gradient-boosted tree classifier on the feature store data, then runs scoring as a batch job that reads from Kudu and writes scores back. In production, this would run on a schedule (every 15–60 minutes).

PySpark — churn_scoring.py 
from pyspark.sql import SparkSession
from pyspark.sql.functions import *
from pyspark.ml.feature import VectorAssembler, StringIndexer
from pyspark.ml.classification import GBTClassifier
from pyspark.ml import Pipeline
import time

spark = SparkSession.builder \
    .appName("Churn Model Scoring") \
    .getOrCreate()

# ── 1. Read current features from Kudu ──
df = spark.read.format("kudu") \
    .option("kudu.master", "KUDU_MASTER:7051") \
    .option("kudu.table", "impala::default.subscriber_features") \
    .load()

print(f"Read {df.count()} subscriber records from Kudu")

# ── 2. Feature engineering ──
# For this lab, we synthesize a churn label based on known risk indicators.
# In production, you'd have historical churn labels from the CRM.
df_labeled = df.withColumn("churn_label",
    when(
        (col("complaint_count_30d") > 3) |
        (col("days_since_recharge") > 45) |
        (col("call_drop_rate_7d") > 0.10) |
        ((col("revenue_trend") == "declining") & (col("nps_score") < 5)),
        1.0
    ).otherwise(0.0)
)

# Select numeric features
feature_cols = [
    "account_age_days", "avg_monthly_revenue", "total_calls_30d",
    "total_data_mb_30d", "complaint_count_30d", "call_drop_rate_7d",
    "days_since_recharge"
]

# Handle nulls
for c in feature_cols:
    df_labeled = df_labeled.fillna({c: 0})

assembler = VectorAssembler(inputCols=feature_cols, outputCol="features")

# ── 3. Train model ──
gbt = GBTClassifier(
    featuresCol="features",
    labelCol="churn_label",
    maxIter=50,
    maxDepth=5,
    seed=42
)

pipeline = Pipeline(stages=[assembler, gbt])

# Split for training (in production, use a proper train/test split
# with historical data, not the current feature store)
train_df, score_df = df_labeled.randomSplit([0.3, 0.7], seed=42)
model = pipeline.fit(train_df)

# ── 4. Score ALL subscribers ──
predictions = model.transform(df_labeled)

# Map predictions to risk tiers
scored = predictions.select(
    "subscriber_id",
    col("probability").getItem(1).alias("churn_score"),
    when(col("probability").getItem(1) > 0.7, "high")
     .when(col("probability").getItem(1) > 0.4, "medium")
     .otherwise("low").alias("churn_risk_tier"),
    current_timestamp().alias("last_scored_at")
)

# ── 5. Write scores back to Kudu ──
scored.write.format("kudu") \
    .option("kudu.master", "KUDU_MASTER:7051") \
    .option("kudu.table", "impala::default.subscriber_features") \
    .option("kudu.operation", "upsert") \
    .mode("append") \
    .save()

# Summary
scored.groupBy("churn_risk_tier").count().show()
print("Scoring complete — scores written back to Kudu")
Shell 
spark-submit \
  --packages org.apache.kudu:kudu-spark3_2.12:1.18.1 \
  --conf spark.executor.memory=4g \
  --conf spark.executor.cores=2 \
  churn_scoring.py

The key pattern here: Spark reads features from Kudu, scores them, and writes scores back to Kudu. The feature store and the scoring output live in the same table. No ETL pipeline between them. No staging area. No batch vs. serving split. One table, always current.

Retention Dashboard Queries

These are the queries the retention team would run through Hue dashboards. Every query hits the single Kudu feature store table — no joins, no ETL lag.

5a. High-Risk Subscribers Not Yet Contacted

Impala SQL 
-- Priority action list for the retention team
SELECT subscriber_id, region, plan_type,
       churn_score, avg_monthly_revenue,
       complaint_count_30d, days_since_recharge,
       call_drop_rate_7d
FROM subscriber_features
WHERE churn_risk_tier = 'high'
  AND retention_contacted = FALSE
ORDER BY churn_score DESC, avg_monthly_revenue DESC
LIMIT 50;

5b. Regional Risk Summary

Impala SQL 
-- Executive view: churn risk distribution by region
SELECT region,
       COUNT(*) AS total_subscribers,
       COUNT(CASE WHEN churn_risk_tier = 'high' THEN 1 END) AS high_risk,
       COUNT(CASE WHEN churn_risk_tier = 'medium' THEN 1 END) AS medium_risk,
       ROUND(AVG(churn_score), 3) AS avg_churn_score,
       ROUND(SUM(CASE WHEN churn_risk_tier = 'high'
                      THEN avg_monthly_revenue ELSE 0 END), 2) AS at_risk_revenue
FROM subscriber_features
WHERE churn_score IS NOT NULL
GROUP BY region
ORDER BY at_risk_revenue DESC;

5c. Retention Campaign Effectiveness

Impala SQL 
-- How are retention offers performing?
SELECT retention_offer,
       COUNT(*) AS total_offered,
       COUNT(CASE WHEN retention_outcome = 'accepted' THEN 1 END) AS accepted,
       COUNT(CASE WHEN retention_outcome = 'declined' THEN 1 END) AS declined,
       ROUND(COUNT(CASE WHEN retention_outcome = 'accepted' THEN 1 END)
             * 100.0 / COUNT(*), 1) AS acceptance_rate_pct
FROM subscriber_features
WHERE retention_contacted = TRUE
  AND retention_offer IS NOT NULL
GROUP BY retention_offer
ORDER BY acceptance_rate_pct DESC;

5d. Simulate a Retention Action

Impala SQL 
-- Mark a subscriber as contacted with an offer
UPSERT INTO subscriber_features
  (subscriber_id, retention_contacted, retention_offer,
   retention_outcome, last_contacted_at)
VALUES
  ('SUB-00042', TRUE, '20% discount 3 months', 'pending', now());

-- Verify
SELECT subscriber_id, churn_score, churn_risk_tier,
       retention_contacted, retention_offer, retention_outcome
FROM subscriber_features
WHERE subscriber_id = 'SUB-00042';

Closed-loop pattern: The retention team's action (contacting a subscriber, recording the offer response) writes back to the same Kudu table. The next scoring cycle picks up the retention_outcome as an additional signal. The model learns what works.
