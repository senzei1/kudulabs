Lab 02

Continuous Churn Prediction

Build a real-time feature store in Kudu, ingest behavioral events via NiFi, score subscribers with a Spark ML model, and serve retention dashboards through Impala — all with sub-minute freshness.

Duration: 120 minutes
Level: Advanced
Components: NiFi, Kudu, Spark 3, Impala, Hue

Secured cluster? Complete Lab 00: Security Setup first. This lab also requires Spark 3 (CDS 3 parcel + CSD), which must be installed separately — it is not bundled with base CDP. Use spark3-submit, not spark-submit.

Prerequisites

  • Lab 00 (Security Setup) completed
  • CDP cluster with NiFi, Kudu, Spark 3, and Impala
  • Spark 3 CSD and parcel installed and activated in CM
  • numpy installed: pip3 install numpy --break-system-packages
  • Your OS user must exist on all YARN NodeManager nodes (sudo useradd -m your_user)
  • Lab 01 completed (familiarity with NiFi → Kudu flow)

Create the Kudu Tables

This lab uses two tables: a feature store (subscriber attributes + model scores) and an events table (raw behavioral signals). The Spark scoring job reads from both and writes scores back to the feature store.

1a. Feature Store Table

Impala SQL (via Hue) 
CREATE TABLE subscriber_features (
  subscriber_id       STRING,
  region              STRING,
  plan_type           STRING,
  account_age_days    INT,
  avg_monthly_revenue DECIMAL(10,2),
  revenue_trend       STRING     COMMENT 'increasing | stable | declining',
  total_calls_30d     INT        DEFAULT 0,
  total_data_mb_30d   DOUBLE     DEFAULT 0,
  complaint_count_30d INT        DEFAULT 0,
  last_complaint_date TIMESTAMP,
  call_drop_rate_7d   FLOAT      DEFAULT 0,
  days_since_recharge INT,
  competitive_exposure BOOLEAN   DEFAULT FALSE,
  nps_score           INT,
  last_nps_date       TIMESTAMP,
  churn_score         FLOAT,
  churn_risk_tier     STRING     COMMENT 'high | medium | low',
  last_scored_at      TIMESTAMP,
  retention_contacted BOOLEAN    DEFAULT FALSE,
  retention_offer     STRING,
  retention_outcome   STRING     COMMENT 'accepted | declined | pending | null',
  last_contacted_at   TIMESTAMP,
  PRIMARY KEY (subscriber_id)
)
PARTITION BY HASH (subscriber_id) PARTITIONS 16
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = '<YOUR-KUDU-MASTER>:7051',
  'kudu.num_tablet_replicas' = '3'
);

Design note: No range partitioning here — this is a current-state table, not a time-series. Each subscriber has exactly one row that gets updated as new signals arrive and scores are recalculated.

1b. Events Table

Raw behavioral events land here from NiFi. The Spark scoring job aggregates them into the feature store. This separation avoids Kudu's full-row UPSERT limitation — events are simple inserts, feature aggregation happens in Spark.

Impala SQL 
CREATE TABLE subscriber_events (
  event_id        STRING,
  subscriber_id   STRING,
  event_time      TIMESTAMP,
  event_type      STRING     COMMENT 'complaint | recharge | network | nps',
  event_value     DOUBLE,
  event_detail    STRING,
  PRIMARY KEY (event_id)
)
PARTITION BY HASH (event_id) PARTITIONS 8
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = '<YOUR-KUDU-MASTER>:7051',
  'kudu.num_tablet_replicas' = '3'
);

Seed the Feature Store

What this script does: Generates 50,000 synthetic subscriber records with randomized behavioral attributes (revenue, complaints, call drop rate, etc.) and writes them to the subscriber_features table via Spark's Kudu connector. This simulates a mature feature store with historical subscriber data. All scoring fields (churn_score, risk_tier) are left null — the scoring model will populate them in Step 4.

Python — seed_features.py 
from decimal import Decimal
from pyspark.sql import SparkSession
from pyspark.sql.types import *
import random

spark = SparkSession.builder \
    .appName("Seed Churn Feature Store") \
    .getOrCreate()

num_subscribers = 50000
regions = ['San Jose', 'Heredia', 'Alajuela', 'Cartago',
           'Guanacaste', 'Puntarenas', 'Limon',
           'Bogota', 'Medellin', 'Lima']
plans = ['Prepaid Basic', 'Prepaid Plus', 'Postpaid Standard',
         'Postpaid Pro', 'Enterprise']
trends = ['increasing', 'stable', 'declining']

data = []
for i in range(num_subscribers):
    sub_id = f"SUB-{i:05d}"
    region = random.choice(regions)
    plan = random.choice(plans)
    age = random.randint(30, 2000)
    revenue = Decimal(str(round(random.uniform(5.0, 150.0), 2)))
    trend = random.choice(trends)
    calls = random.randint(0, 500)
    data_mb = round(random.uniform(0, 50000), 1)
    complaints = random.randint(0, 8)
    drop_rate = round(random.uniform(0, 0.15), 4)
    recharge_days = random.randint(0, 90)
    nps = random.choice([None, random.randInt(1, 10)])

    data.append((sub_id, region, plan, age, revenue, trend,
                 calls, data_mb, complaints, None, drop_rate,
                 recharge_days, False, nps, None,
                 None, None, None,
                 False, None, None, None))

schema = StructType([
    StructField("subscriber_id", StringType()),
    StructField("region", StringType()),
    StructField("plan_type", StringType()),
    StructField("account_age_days", IntegerType()),
    StructField("avg_monthly_revenue", DecimalType(10,2)),
    StructField("revenue_trend", StringType()),
    StructField("total_calls_30d", IntegerType()),
    StructField("total_data_mb_30d", DoubleType()),
    StructField("complaint_count_30d", IntegerType()),
    StructField("last_complaint_date", TimestampType()),
    StructField("call_drop_rate_7d", FloatType()),
    StructField("days_since_recharge", IntegerType()),
    StructField("competitive_exposure", BooleanType()),
    StructField("nps_score", IntegerType()),
    StructField("last_nps_date", TimestampType()),
    StructField("churn_score", FloatType()),
    StructField("churn_risk_tier", StringType()),
    StructField("last_scored_at", TimestampType()),
    StructField("retention_contacted", BooleanType()),
    StructField("retention_offer", StringType()),
    StructField("retention_outcome", StringType()),
    StructField("last_contacted_at", TimestampType()),
])

df = spark.createDataFrame(data, schema)

df.write.format("kudu") \
    .option("kudu.master", "<YOUR-KUDU-MASTER>:7051") \
    .option("kudu.table", "impala::default.subscriber_features") \
    .option("kudu.operation", "upsert") \
    .mode("append") \
    .save()

print(f"Seeded {num_subscribers} subscribers")
Shell 
spark3-submit \
  --master local[2] \
  --packages org.apache.kudu:kudu-spark3_2.12:1.18.1 \
  --driver-memory 1g \
  seed_features.py

--master local[2] runs Spark in local mode, bypassing YARN. On resource-constrained lab clusters, YARN may not have enough capacity to allocate executors. Local mode is fine for 50K rows. For production workloads, use --master yarn.

Build the NiFi Event Ingestion Flow

This NiFi flow simulates behavioral events arriving from multiple source systems — CRM (complaints), billing (recharges), network monitoring (call drop rates), and NPS surveys. Each event type is routed separately for visibility, then all land in the subscriber_events table.

In production: the ExecuteScript generator would be replaced by ConsumeKafkaRecord processors reading from real Kafka topics fed by the CRM, billing system, and network monitoring tools. The routing and PutKudu steps remain the same.

This script generates random behavioral events tagged with an event_type attribute. Each event simulates a signal from a different source system.

Groovy — Event Generator Script Body 
import groovy.json.JsonOutput
import org.apache.nifi.processor.io.StreamCallback
import java.nio.charset.StandardCharsets

def flowFile = session.create()
def random = new Random()
def fmt = "yyyy-MM-dd HH:mm:ss"
def now = new Date().format(fmt)

def types = ['complaint', 'recharge', 'network', 'nps']
def eventType = types[random.nextInt(types.size())]
def subId = String.format("SUB-%05d", random.nextInt(50000))

def event = [
  event_id      : UUID.randomUUID().toString(),
  subscriber_id : subId,
  event_time    : now,
  event_type    : eventType,
  event_value   : null,
  event_detail  : null
]

switch (eventType) {
    case 'complaint':
        event.event_value = 1.0
        def cats = ['billing_dispute', 'service_outage',
                    'poor_coverage', 'dropped_call']
        event.event_detail = cats[random.nextInt(4)]
        break
    case 'recharge':
        event.event_value = Math.round(
          random.nextDouble() * 4500 + 500) / 100.0
        event.event_detail = 'prepaid_topup'
        break
    case 'network':
        event.event_value = Math.round(
          random.nextFloat() * 0.20f * 10000) / 10000.0
        event.event_detail = 'call_drop_rate'
        break
    case 'nps':
        event.event_value = random.nextInt(10) + 1
        event.event_detail = 'survey_response'
        break
}

flowFile = session.write(flowFile, { inputStream, outputStream ->
    outputStream.write(
      JsonOutput.toJson(event).getBytes(StandardCharsets.UTF_8))
} as StreamCallback)

flowFile = session.putAttribute(flowFile, 'mime.type',
                                'application/json')
flowFile = session.putAttribute(flowFile, 'event_type', eventType)
session.transfer(flowFile, REL_SUCCESS)

Build the Churn Scoring Model (Spark 3 ML)

What this script does — four phases:

Phase 1 — Read features from Kudu. Pulls all 50,000 subscriber records from subscriber_features with their behavioral attributes.

Phase 2 — Create a synthetic churn label. Since we don't have real historical churn data, the script flags subscribers as "churned" based on known risk indicators: high complaint count (>3), long time since recharge (>45 days), high call drop rate (>10%), or declining revenue with low NPS. In production, this label would come from actual CRM data.

Phase 3 — Train a GBT classifier. Gradient Boosted Trees learns which behavioral patterns predict churn. Then scores ALL subscribers with a probability between 0 and 1.

Phase 4 — Write scores back to Kudu. Each subscriber gets a churn_score, churn_risk_tier (high/medium/low), and last_scored_at timestamp — UPSERTed into the same feature store table.

The Kudu pattern: Read from Kudu → process → write back to the same Kudu table. The feature store and the scoring output live in one place. No separate serving layer, no ETL between them. Impala dashboards see fresh scores immediately after each scoring run.

Python — churn_scoring.py 
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, when, current_timestamp
from pyspark.sql.types import FloatType
from pyspark.ml.feature import VectorAssembler
from pyspark.ml.classification import GBTClassifier
from pyspark.ml.functions import vector_to_array
from pyspark.ml import Pipeline

spark = SparkSession.builder \
    .appName("Churn Model Scoring") \
    .getOrCreate()

# ── Phase 1: Read features from Kudu ──
df = spark.read.format("kudu") \
    .option("kudu.master", "<YOUR-KUDU-MASTER>:7051") \
    .option("kudu.table", "impala::default.subscriber_features") \
    .load()

print(f"Read {df.count()} subscriber records from Kudu")

# ── Phase 2: Create synthetic churn label ──
df_labeled = df.withColumn("churn_label",
    when(
        (col("complaint_count_30d") > 3) |
        (col("days_since_recharge") > 45) |
        (col("call_drop_rate_7d") > 0.10) |
        ((col("revenue_trend") == "declining") &
         (col("nps_score") < 5)),
        1.0
    ).otherwise(0.0)
)

feature_cols = [
    "account_age_days", "avg_monthly_revenue", "total_calls_30d",
    "total_data_mb_30d", "complaint_count_30d", "call_drop_rate_7d",
    "days_since_recharge"
]

for c in feature_cols:
    df_labeled = df_labeled.fillna({c: 0})

# Cast Decimal to Double for ML compatibility
df_labeled = df_labeled.withColumn(
    "avg_monthly_revenue",
    col("avg_monthly_revenue").cast("double"))

assembler = VectorAssembler(
    inputCols=feature_cols, outputCol="features")

# ── Phase 3: Train and score ──
gbt = GBTClassifier(
    featuresCol="features",
    labelCol="churn_label",
    maxIter=50, maxDepth=5, seed=42)

pipeline = Pipeline(stages=[assembler, gbt])

train_df, _ = df_labeled.randomSplit([0.3, 0.7], seed=42)
model = pipeline.fit(train_df)
predictions = model.transform(df_labeled)

# ── Phase 4: Write scores back to Kudu ──
scored = predictions.select(
    "subscriber_id",
    vector_to_array("probability").getItem(1)
        .cast(FloatType()).alias("churn_score"),
    when(vector_to_array("probability").getItem(1) > 0.7, "high")
     .when(vector_to_array("probability").getItem(1) > 0.4, "medium")
     .otherwise("low").alias("churn_risk_tier"),
    current_timestamp().alias("last_scored_at")
)

scored.write.format("kudu") \
    .option("kudu.master", "<YOUR-KUDU-MASTER>:7051") \
    .option("kudu.table", "impala::default.subscriber_features") \
    .option("kudu.operation", "upsert") \
    .mode("append") \
    .save()

scored.groupBy("churn_risk_tier").count().show()
print("Scoring complete — scores written back to Kudu")
Shell 
spark3-submit \
  --master local[2] \
  --packages org.apache.kudu:kudu-spark3_2.12:1.18.1 \
  --driver-memory 1g \
  churn_scoring.py

Retention Dashboard Queries

These queries hit the single Kudu feature store table — no joins, no ETL lag. Run each query separately in Hue (not all at once).

5a. High-Risk Subscribers Not Yet Contacted

SELECT subscriber_id, region, plan_type,
       churn_score, avg_monthly_revenue,
       complaint_count_30d, days_since_recharge,
       call_drop_rate_7d
FROM subscriber_features
WHERE churn_risk_tier = 'high'
  AND retention_contacted = FALSE
ORDER BY churn_score DESC, avg_monthly_revenue DESC
LIMIT 50;

5b. Regional Risk Summary

SELECT region,
       COUNT(*) AS total_subscribers,
       COUNT(CASE WHEN churn_risk_tier = 'high' THEN 1 END) AS high_risk,
       COUNT(CASE WHEN churn_risk_tier = 'medium' THEN 1 END) AS medium_risk,
       ROUND(AVG(churn_score), 3) AS avg_churn_score
FROM subscriber_features
WHERE churn_score IS NOT NULL
GROUP BY region
ORDER BY high_risk DESC;

5d. Retention Campaign Effectiveness

Run this after the retention simulation above: 

SELECT retention_offer,
       COUNT(*) AS total_offered,
       COUNT(CASE WHEN retention_outcome = 'accepted' THEN 1 END) AS accepted,
       COUNT(CASE WHEN retention_outcome = 'declined' THEN 1 END) AS declined,
       ROUND(COUNT(CASE WHEN retention_outcome = 'accepted' THEN 1 END)
             * 100.0 / COUNT(*), 1) AS acceptance_rate_pct
FROM subscriber_features
WHERE retention_contacted = TRUE
  AND retention_offer IS NOT NULL
GROUP BY retention_offer
ORDER BY acceptance_rate_pct DESC;

Closed-loop pattern: The retention team's actions write back to the same Kudu table. The next scoring cycle picks up retention_outcome as an additional signal. The model learns what works.
