Lab 03

CDC Inventory & Warehouse Tracking

Capture real-time changes from a PostgreSQL inventory database, replicate to Kudu for instant analytical visibility, run replenishment queries in Impala, and age historical snapshots to Iceberg. A complete tiered-storage architecture.

Prerequisites

  • Lab 00 (Security Setup) completed

Set Up the PostgreSQL Source

Create the operational inventory database in PostgreSQL. This simulates an ERP/WMS system that tracks stock levels in real time.

1a. Install PostgreSQL (if needed)

# RHEL/Rocky 9
sudo dnf install -y postgresql16-server --disablerepo=pgdg13
sudo /usr/pgsql-16/bin/postgresql-16-setup initdb
sudo systemctl enable --now postgresql-16

1b. Enable Remote Connections

By default, PostgreSQL only listens on localhost. NiFi (which may run on a different node) needs to connect remotely. 

# Edit postgresql.conf
sudo vi /var/lib/pgsql/16/data/postgresql.conf
# Change: listen_addresses = 'localhost'
# To:     listen_addresses = '*'

# Allow remote connections in pg_hba.conf
echo "host    all    all    0.0.0.0/0    md5" | \
  sudo tee -a /var/lib/pgsql/16/data/pg_hba.conf

# Set a password for the postgres user
sudo -u postgres psql -c "ALTER USER postgres PASSWORD 'postgres';"

# Restart
sudo systemctl restart postgresql-16

# Verify it's listening on all interfaces
netstat -tlnp | grep 5432
# Should show 0.0.0.0:5432, not 127.0.0.1:5432

Create the Kudu Target Tables

Impala SQL (via Hue) 
CREATE TABLE inventory_realtime (
  warehouse_id      STRING,
  sku               STRING,
  location_code     STRING,
  product_name      STRING,
  category          STRING,
  quantity_on_hand  INT,
  quantity_reserved INT,
  quantity_available INT    COMMENT 'computed: on_hand - reserved',
  unit_cost         DECIMAL(10,2),
  supplier_id       STRING,
  last_movement     STRING,
  last_updated      TIMESTAMP,
  cdc_operation     STRING  COMMENT 'INSERT | UPDATE | DELETE',
  cdc_timestamp     TIMESTAMP,
  PRIMARY KEY (warehouse_id, sku, location_code)
)
PARTITION BY
  HASH (warehouse_id) PARTITIONS 4,
  HASH (sku) PARTITIONS 4
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = '<YOUR-KUDU-MASTER>:7051',
  'kudu.num_tablet_replicas' = '3'
);

If Iceberg is available, also create the cold-tier table:

CREATE TABLE inventory_history (
  warehouse_id      STRING,
  sku               STRING,
  location_code     STRING,
  product_name      STRING,
  category          STRING,
  quantity_on_hand  INT,
  quantity_reserved INT,
  unit_cost         DECIMAL(10,2),
  supplier_id       STRING,
  last_movement     STRING,
  snapshot_date     DATE,
  snapshot_timestamp TIMESTAMP
)
STORED AS ICEBERG;

Build the NiFi CDC Flow

This flow uses QueryDatabaseTableRecord for polling-based CDC. It polls the PostgreSQL inventory table on a schedule, using last_updated as the high-water-mark column to capture only changed rows since the last poll.

3a. Controller Services

Create and enable these controller services (reuse from previous labs where possible):

Service Key Settings
DBCPConnectionPool Database Connection URL: jdbc:postgresql://<PG-HOST>:5432/warehouse_ops
Database Driver Class Name: org.postgresql.Driver
Database Driver Location(s): /opt/cloudera/cm/lib/postgresql-42.7.2.jar
Database User: postgres
Password: postgres (or your password)
JsonTreeReader Schema Access Strategy: Infer Schema
Timestamp Format: yyyy-MM-dd HH:mm:ss
JsonRecordSetWriter Schema Write Strategy: Do Not Write Schema
KeytabCredentialsService Kerberos Principal: nifi/_HOST@<YOUR-REALM>
Kerberos Keytab: (path from Lab 00 Step 5)

JDBC driver path: If NiFi can't read /opt/cloudera/cm/lib/postgresql-42.7.2.jar due to filesystem restrictions, copy it to /tmp/postgresql-42.7.2.jar and reference that path instead.

3c. UpdateRecord — Add CDC Metadata

Add CDC metadata fields and compute the available quantity. UpdateRecord requires both a Record Reader and Record Writer.

Property Value
Record Reader JsonTreeReader
Record Writer JsonRecordSetWriter

Add these dynamic properties (click +):

Property Name Value
/cdc_operation UPSERT
/cdc_timestamp ${now():format('yyyy-MM-dd HH:mm:ss')}
/quantity_available ${field.value:minus(/quantity_reserved)}

quantity_available computation: The expression above attempts to compute on_hand - reserved. If NiFi Expression Language doesn't evaluate correctly for your version, set a static value like 0 and compute it in Impala instead: SELECT *, quantity_on_hand - quantity_reserved AS available FROM inventory_realtime

3d. PutKudu — Write to Kudu

Property Value
Kudu Masters <YOUR-KUDU-MASTER>:7051
Table Name impala::default.inventory_realtime
Record Reader JsonTreeReader
Kudu Operation Type UPSERT
Flush Mode AUTO_FLUSH_BACKGROUND
Kerberos Credentials Service KeytabCredentialsService

Connect: QueryDatabaseTableRecord → UpdateRecord → PutKudu. Route PutKudu failure to LogAttribute; auto-terminate success.

Generate Warehouse Activity & Observe Replication

Simulate real warehouse operations in PostgreSQL and watch them replicate to Kudu within seconds (one polling cycle).

4a. Bulk Activity Generator

Run this script to generate 500 inventory movements (inserts and updates): 

sudo -u postgres psql -d warehouse_ops << 'SQL'
DO $$
DECLARE
  i INT;
  wh TEXT;
  warehouses TEXT[] := ARRAY['WH-SJO-01','WH-BOG-01','WH-LIM-01'];
  movements TEXT[] := ARRAY['pick','receipt','putaway','adjustment'];
BEGIN
  FOR i IN 1..500 LOOP
    wh := warehouses[1 + floor(random() * 3)::int];
    INSERT INTO inventory VALUES (
      wh,
      'SKU-' || lpad((100 + floor(random()*200))::text, 3, '0'),
      chr(65 + floor(random()*5)::int) || '-' ||
        lpad(floor(random()*10)::text,2,'0') || '-' ||
        lpad(floor(random()*10)::text,2,'0'),
      'Product ' || i,
      (ARRAY['Electronics','Hardware','Accessories','IoT'])
        [1 + floor(random()*4)::int],
      floor(random() * 1000)::int,
      floor(random() * 100)::int,
      round((random() * 100 + 1)::numeric, 2),
      'SUPP-' || lpad(floor(random()*10+1)::text, 2, '0'),
      movements[1 + floor(random()*4)::int],
      NOW()
    )
    ON CONFLICT (warehouse_id, sku, location_code)
    DO UPDATE SET
      quantity_on_hand = EXCLUDED.quantity_on_hand,
      last_movement = EXCLUDED.last_movement,
      last_updated = NOW();
  END LOOP;
END $$;
SELECT COUNT(*) AS total_rows FROM inventory;
SQL

4b. Continuous Activity Simulator

Run this in a separate terminal to simulate ongoing warehouse picks every 30 seconds: 

while true; do
  sudo -u postgres psql -d warehouse_ops -c "
    UPDATE inventory
    SET quantity_on_hand = quantity_on_hand - floor(random()*10)::int,
        last_movement = 'pick',
        last_updated = NOW()
    WHERE ctid IN (
      SELECT ctid FROM inventory ORDER BY random() LIMIT 20
    );
  "
  echo "$(date) - Updated 20 random rows"
  sleep 30
done

Wait 10–15 seconds (one CDC polling cycle), then verify changes replicated to Kudu: 

SELECT warehouse_id, sku, quantity_on_hand,
       last_movement, cdc_timestamp
FROM inventory_realtime
ORDER BY cdc_timestamp DESC
LIMIT 10;

Real-Time Replenishment & Analytics Queries

5a. Low-Stock Alert

SELECT warehouse_id, sku, product_name, category,
       quantity_on_hand,
       CASE
         WHEN quantity_on_hand < 20 THEN 'CRITICAL'
         WHEN quantity_on_hand < 50 THEN 'LOW'
         ELSE 'ADEQUATE'
       END AS stock_status
FROM inventory_realtime
WHERE quantity_on_hand < 100
ORDER BY quantity_on_hand ASC;

5b. Inventory Value by Warehouse

SELECT warehouse_id,
       COUNT(DISTINCT sku) AS unique_skus,
       SUM(quantity_on_hand) AS total_units,
       ROUND(CAST(SUM(quantity_on_hand * unit_cost) AS DOUBLE), 2)
         AS total_value
FROM inventory_realtime
GROUP BY warehouse_id
ORDER BY total_value DESC;

5c. Supplier Concentration Risk

SELECT supplier_id,
       COUNT(DISTINCT sku) AS skus_supplied,
       COUNT(DISTINCT warehouse_id) AS warehouses_served,
       ROUND(CAST(SUM(quantity_on_hand * unit_cost) AS DOUBLE), 2)
         AS total_value_at_risk
FROM inventory_realtime
GROUP BY supplier_id
ORDER BY total_value_at_risk DESC;

Contrast with the alternative: These queries run in sub-seconds on Kudu because they're columnar scans reading only the needed columns. With the traditional PostgreSQL-export-to-Hive pattern, this data would be hours stale. With HBase, these GROUP BY queries would require full-table scans deserializing every column.

✓ Checkpoint

  • Low-stock alert returns items needing reorder
  • Inventory value query completes sub-second
  • All queries reflect recent changes from the continuous simulator

Age Historical Data to Iceberg (Cold Tier)

Requires Iceberg. If Iceberg is not installed on your cluster, skip this step — Steps 1–5 demonstrate the CDC-to-Kudu pipeline standalone.

What this script does: Takes a daily snapshot of the current Kudu inventory state and appends it to an Iceberg table. This preserves historical inventory levels for trend analysis, auditing, and regulatory reporting — without keeping months of data in Kudu.

Python — inventory_snapshot.py 
from pyspark.sql import SparkSession
from pyspark.sql.functions import current_date, current_timestamp

spark = SparkSession.builder \
    .appName("Inventory Daily Snapshot") \
    .config("spark.sql.catalog.iceberg_catalog",
            "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.iceberg_catalog.type", "hive") \
    .config("spark.sql.iceberg.handle-timestamp-without-timezone",
            "true") \
    .getOrCreate()

# Read current state from Kudu
current = spark.read.format("kudu") \
    .option("kudu.master", "<YOUR-KUDU-MASTER>:7051") \
    .option("kudu.table", "impala::default.inventory_realtime") \
    .load() \
    .select(
        "warehouse_id", "sku", "location_code", "product_name",
        "category", "quantity_on_hand", "quantity_reserved",
        "unit_cost", "supplier_id", "last_movement"
    ) \
    .withColumn("snapshot_date", current_date()) \
    .withColumn("snapshot_timestamp", current_timestamp())

# Append to Iceberg historical table
current.writeTo("iceberg_catalog.default.inventory_history") \
    .append()

print(f"Snapshot written: {current.count()} rows "
      f"as of {current.first().snapshot_date}")
Shell 
spark3-submit \
  --master local[2] \
  --packages org.apache.kudu:kudu-spark3_2.12:1.18.1 \
  --driver-memory 1g \
  inventory_snapshot.py

Query Across Both Tiers

-- Current state (Kudu — hot tier)
SELECT warehouse_id, sku, quantity_on_hand
FROM inventory_realtime
WHERE warehouse_id = 'WH-SJO-01';

-- Historical snapshots (Iceberg — cold tier)
SELECT snapshot_date, COUNT(*) AS rows_archived
FROM inventory_history
GROUP BY snapshot_date;

Impala queries both tiers transparently. The operational team queries Kudu for current stock. The analytics team queries Iceberg for trends. No Lambda architecture. No separate ETL pipeline. Just a daily snapshot job.

Troubleshooting

Error Cause Fix
Connection refused to PostgreSQL PostgreSQL listening on localhost only Set listen_addresses = '*' in postgresql.conf, add remote host entry in pg_hba.conf, restart PostgreSQL
SCRAM-based authentication, but no password was provided No password set on PostgreSQL user ALTER USER postgres PASSWORD 'postgres'; then set password in NiFi's DBCPConnectionPool
table does not exist: impala::default.inventory_realtime Kudu table not created yet, or PutKudu cached stale state Create the table in Impala, then stop and restart PutKudu
IllegalTypeConversionException: Timestamp Long parsing failed JsonTreeReader missing timestamp format Set Timestamp Format to yyyy-MM-dd HH:mm:ss on JsonTreeReader
Cannot handle timestamp without timezone (Iceberg) Spark Iceberg connector requires explicit timezone config Add --conf spark.sql.iceberg.handle-timestamp-without-timezone=true to spark3-submit
Permission denied: user does not have [SELECT] privilege Missing Ranger Hadoop SQL policy for Iceberg tables Add your user to Ranger Hive/SQL policy with all permissions on default database
NiFi can't read JDBC driver jar Filesystem access restriction Copy driver to /tmp/postgresql-42.7.2.jar and reference that path
NonRecoverableException: Unauthorized action nifi principal lacks Kudu permissions Add Ranger Kudu policy granting nifi all permissions on target table
