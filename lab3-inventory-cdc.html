Lab 03

CDC Inventory & Warehouse Tracking

Capture real-time changes from a PostgreSQL inventory database, replicate to Kudu for instant analytical visibility, run replenishment queries in Impala, and age historical snapshots to Iceberg. A complete tiered-storage architecture.

Prerequisites

  • CDP cluster with NiFi, Kudu, Impala, Spark, and Iceberg
  • PostgreSQL 12+ instance with logical replication enabled (wal_level = logical)
  • PostgreSQL JDBC driver available to NiFi
  • NiFi user with access to deploy CDC processors

Set Up the PostgreSQL Source

Create the operational inventory database in PostgreSQL. This simulates an ERP/WMS system that tracks stock levels in real time.

1a. Enable Logical Replication

postgresql.conf 
# Required for CDC
wal_level = logical
max_replication_slots = 4
max_wal_senders = 4
Shell 
# Restart PostgreSQL after config change
sudo systemctl restart postgresql

1b. Create the Inventory Schema

PostgreSQL

\c warehouse_ops;

  warehouse_id    VARCHAR(20),
  sku             VARCHAR(30),
  location_code   VARCHAR(20),

Create the Kudu Target Table

Impala SQL 
CREATE TABLE inventory_realtime (
  warehouse_id      STRING,
  sku               STRING,
  location_code     STRING,
  product_name      STRING,
  category          STRING,
  quantity_on_hand  INT,
  quantity_reserved INT,
  quantity_available INT    COMMENT 'computed: on_hand - reserved',
  unit_cost         DECIMAL(10,2),
  supplier_id       STRING,
  last_movement     STRING,
  last_updated      TIMESTAMP,
  cdc_operation     STRING  COMMENT 'INSERT | UPDATE | DELETE',
  cdc_timestamp     TIMESTAMP,
  PRIMARY KEY (warehouse_id, sku, location_code)
)
PARTITION BY
  HASH (warehouse_id) PARTITIONS 4,
  HASH (sku) PARTITIONS 4
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = 'KUDU_MASTER:7051',
  'kudu.num_tablet_replicas' = '3'
);

-- Also create the Iceberg cold-tier table for historical snapshots
CREATE TABLE inventory_history (
  warehouse_id      STRING,
  sku               STRING,
  location_code     STRING,
  product_name      STRING,
  category          STRING,
  quantity_on_hand  INT,
  quantity_reserved INT,
  unit_cost         DECIMAL(10,2),
  supplier_id       STRING,
  last_movement     STRING,
  snapshot_date     DATE,
  snapshot_timestamp TIMESTAMP
)
STORED AS ICEBERG;

Build the NiFi CDC Flow

NiFi's CaptureChangeMySQL or QueryDatabaseTableRecord processors handle CDC. For PostgreSQL, we use either the CaptureChangePostgres processor (if available in your NiFi version) or a polling-based CDC with QueryDatabaseTableRecord.

3a. Option A: Polling-Based CDC (QueryDatabaseTableRecord)

This approach polls the source table on a schedule, using a high-water-mark column (last_updated) to capture only changed rows.

Simulate Warehouse Operations & Observe Real-Time Replication

Now simulate real warehouse activity in PostgreSQL and watch it replicate to Kudu within seconds.

4a. Pick Operation (Reduce Stock)

PostgreSQL 
-- A warehouse picker fulfills an order
UPDATE inventory
SET quantity_on_hand = quantity_on_hand - 15,
    quantity_reserved = quantity_reserved - 15,
    last_movement = 'pick',
    last_updated = NOW()
WHERE warehouse_id = 'WH-SJO-01'
  AND sku = 'SKU-001'
  AND location_code = 'A-01-01';

Wait 10–15 seconds (one CDC polling cycle), then verify in Impala:

Impala SQL 
SELECT warehouse_id, sku, quantity_on_hand, quantity_reserved,
       quantity_available, last_movement, cdc_timestamp
FROM inventory_realtime
WHERE warehouse_id = 'WH-SJO-01' AND sku = 'SKU-001';

4b. Receipt Operation (Restock)

PostgreSQL 
-- New shipment arrives from supplier
UPDATE inventory
SET quantity_on_hand = quantity_on_hand + 500,
    last_movement = 'receipt',
    last_updated = NOW()
WHERE warehouse_id = 'WH-BOG-01'
  AND sku = 'SKU-004'
  AND location_code = 'C-03-01';

Real-Time Replenishment & Analytics Queries

5a. Low-Stock Alert Query

Impala SQL 
-- Items that need reorder across all warehouses
SELECT warehouse_id, sku, product_name, category,
       quantity_available, unit_cost,
       CASE
         WHEN quantity_available < 20 THEN 'CRITICAL'
         WHEN quantity_available < 50 THEN 'LOW'
         ELSE 'ADEQUATE'
       END AS stock_status
FROM inventory_realtime
WHERE quantity_available < 100
ORDER BY quantity_available ASC;

5b. Inventory Value by Warehouse

Impala SQL 
-- Total inventory value per warehouse (for financial reporting)
SELECT warehouse_id,
       COUNT(DISTINCT sku) AS unique_skus,
       SUM(quantity_on_hand) AS total_units,
       ROUND(SUM(quantity_on_hand * unit_cost), 2) AS total_value,
       ROUND(SUM(quantity_reserved * unit_cost), 2) AS reserved_value
FROM inventory_realtime
GROUP BY warehouse_id
ORDER BY total_value DESC;

5c. Supplier Concentration Risk

Impala SQL 
-- How dependent are we on each supplier?
SELECT supplier_id,
       COUNT(DISTINCT sku) AS skus_supplied,
       COUNT(DISTINCT warehouse_id) AS warehouses_served,
       ROUND(SUM(quantity_on_hand * unit_cost), 2) AS total_value_at_risk
FROM inventory_realtime
GROUP BY supplier_id
ORDER BY total_value_at_risk DESC;

Contrast with the alternative: These queries run in sub-seconds on Kudu because they're columnar scans reading only the needed columns. With the traditional PostgreSQL-export-to-Hive pattern, this data would be hours stale. With HBase, these GROUP BY queries would require full-table scans deserializing every column.

Age Historical Data to Iceberg (Cold Tier)

Take a daily snapshot of the Kudu inventory state and write it to Iceberg. This preserves historical inventory levels for trend analysis, auditing, and regulatory reporting — without keeping months of data in Kudu.

PySpark — inventory_snapshot.py 
from pyspark.sql import SparkSession
from pyspark.sql.functions import current_date, current_timestamp

spark = SparkSession.builder \
    .appName("Inventory Daily Snapshot") \
    .config("spark.sql.catalog.iceberg_catalog", "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.iceberg_catalog.type", "hive") \
    .getOrCreate()

# Read current state from Kudu
current = spark.read.format("kudu") \
    .option("kudu.master", "KUDU_MASTER:7051") \
    .option("kudu.table", "impala::default.inventory_realtime") \
    .load() \
    .select(
        "warehouse_id", "sku", "location_code", "product_name",
        "category", "quantity_on_hand", "quantity_reserved",
        "unit_cost", "supplier_id", "last_movement"
    ) \
    .withColumn("snapshot_date", current_date()) \
    .withColumn("snapshot_timestamp", current_timestamp())

# Append to Iceberg historical table
current.writeTo("iceberg_catalog.default.inventory_history") \
    .append()

print(f"Snapshot written: {current.count()} rows as of {current.first().snapshot_date}")
Shell 
# Run as a daily scheduled job (cron, Oozie, Airflow)
spark-submit \
  --packages org.apache.kudu:kudu-spark3_2.12:1.18.1,org.apache.iceberg:iceberg-spark-runtime-3.3_2.12:1.4.3 \
  inventory_snapshot.py

Query Across Both Tiers

Impala SQL 
-- Current state (Kudu — hot tier)
SELECT warehouse_id, sku, quantity_on_hand
FROM inventory_realtime
WHERE warehouse_id = 'WH-SJO-01';

-- Historical trend (Iceberg — cold tier)
SELECT snapshot_date, sku,
       quantity_on_hand AS stock_level
FROM inventory_history
WHERE warehouse_id = 'WH-SJO-01'
  AND sku = 'SKU-001'
ORDER BY snapshot_date;

Impala queries both tiers transparently. The operational team queries Kudu for current stock. The analytics team queries Iceberg for trends. No Lambda architecture. No separate ETL pipeline. Just a daily snapshot job.
