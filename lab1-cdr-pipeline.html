Create the Kudu Table via Impala

Open Hue or impala-shell and create the CDR table. The schema uses hash partitioning on subscriber_id for write distribution and range partitioning on call_start_time for time-based queries and data lifecycle management.

Impala SQL (via Hue or impala-shell) 
CREATE TABLE cdr_records (
  subscriber_id   STRING,
  call_start_time TIMESTAMP,
  call_end_time   TIMESTAMP,
  duration_sec    INT,
  call_type       STRING     COMMENT 'voice | sms | data',
  called_number   STRING,
  orig_cell_id    STRING,
  term_cell_id    STRING,
  rated_amount    DECIMAL(12,4),
  plan_name       STRING,
  fraud_score     FLOAT,
  fraud_flag      BOOLEAN,
  enriched        BOOLEAN    DEFAULT FALSE,
  PRIMARY KEY (subscriber_id, call_start_time)
)
PARTITION BY
  HASH (subscriber_id) PARTITIONS 16,
  RANGE (call_start_time) (
    PARTITION '2026-04-01' <= VALUES < '2026-05-01',
    PARTITION '2026-05-01' <= VALUES < '2026-06-01',
    PARTITION '2026-06-01' <= VALUES < '2026-07-01'
  )
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = '<YOUR-KUDU-MASTER>:7051',
  'kudu.num_tablet_replicas' = '3'
);

Why 16 hash buckets? Target at least 1 tablet per CPU core across your tablet servers. With 3 tablet servers × 8 cores = 24 cores, 16 hash × 3 range = 48 tablets — about 2 per core. Adjust for your cluster.

Verify the table was created:

DESCRIBE FORMATTED cdr_records;
SHOW TABLE STATS cdr_records;

✓ Checkpoint

You should see the table with 48 tablets (16 hash × 3 range), replication factor 3.

Create the Kafka Topic

Create a topic for the CDR stream. All Kafka CLI commands require --command-config and the environment variables from Lab 00.

Shell 
kafka-topics --create \
  --bootstrap-server <YOUR-KAFKA-BROKER>:9093 \
  --command-config /tmp/kafka-client.properties \
  --topic cdr-raw \
  --partitions 2 \
  --replication-factor 3 \
  --config retention.ms=86400000

# Verify
kafka-topics --describe \
  --bootstrap-server <YOUR-KAFKA-BROKER>:9093 \
  --command-config /tmp/kafka-client.properties \
  --topic cdr-raw

Why only 2 partitions? Each Kafka partition opens an SSL connection that consumes JVM direct buffer memory. On resource-constrained lab clusters, 8 partitions can exhaust direct memory and cause OOM errors on the NiFi consumer. Use 2 for labs; increase for production.

Build the NiFi Flow — CDR Generator

In a real deployment, CDRs arrive from network mediation platforms. For this lab, we generate realistic CDRs using a Groovy script in an ExecuteScript processor.

3a. Controller Services (create these first)

Before building the flow, create and enable these controller services from the NiFi canvas gear icon → Controller Services → +:

Service Configuration
JsonTreeReader Schema Access Strategy: Infer Schema
Timestamp Format: yyyy-MM-dd HH:mm:ss
JsonRecordSetWriter Schema Write Strategy: Inherit Record Schema
KeytabCredentialsService Kerberos Principal: nifi/_HOST@<YOUR-REALM>
Kerberos Keytab: /run/cloudera-scm-agent/process/<PROCESS-ID>-nifi-NIFI_NODE/nifi.keytab
(See Lab 00 Step 5 to find the keytab path on your NiFi node)

Enable all three services before proceeding.

Timestamp Format is critical. Without it, PutKudu will fail with IllegalTypeConversionException: Convert Field Name [call_start_time] to Timestamp Long parsing failed. The reader must know how to parse timestamp strings into epoch microseconds for Kudu.

3b. ExecuteScript — CDR Generator

Add an ExecuteScript processor to the canvas with these settings:

Property Value
Script Engine Groovy
Script Body (see below)

Scheduling tab: Run Schedule = 100 ms (adjust to control ingest rate).

Groovy — CDR Generator Script Body 
import org.apache.nifi.processor.io.StreamCallback
import groovy.json.JsonOutput
import java.nio.charset.StandardCharsets

def flowFile = session.create()

def random = new Random()
def subId = String.format("SUB-%05d", random.nextInt(10000))
def now = new Date()
def duration = random.nextInt(600)
def endTime = new Date(now.time + duration * 1000)
def types = ['voice', 'sms', 'data']
def fmt = "yyyy-MM-dd HH:mm:ss"

def cdr = [
  subscriber_id  : subId,
  call_start_time: now.format(fmt),
  call_end_time  : endTime.format(fmt),
  duration_sec   : duration,
  call_type      : types[random.nextInt(3)],
  called_number  : "+506" + String.format("%08d",
                    random.nextInt(90000000) + 10000000),
  orig_cell_id   : String.format("CELL-%04d", random.nextInt(500)),
  term_cell_id   : String.format("CELL-%04d", random.nextInt(500)),
  rated_amount   : null,
  plan_name      : null,
  fraud_score    : null,
  fraud_flag     : null,
  enriched       : false
]

flowFile = session.write(flowFile, { inputStream, outputStream ->
  outputStream.write(
    JsonOutput.toJson(cdr).getBytes(StandardCharsets.UTF_8))
} as StreamCallback)

flowFile = session.putAttribute(flowFile, 'mime.type',
                                'application/json')
session.transfer(flowFile, REL_SUCCESS)

Dynamic Property Value

Direct buffer memory: Kafka SSL connections consume JVM direct buffer memory. On resource-constrained clusters, the default buffer sizes can exhaust direct memory and cause OutOfMemoryError: Direct buffer memory on the consumer. The dynamic properties above reduce buffer allocation by ~90%. For production clusters with adequate memory, these overrides are not necessary.

4b. ExecuteScript — Fraud Scoring & Rating Engine

This script simulates a realistic enrichment pipeline: it scores each CDR for fraud risk (short calls and premium-rate numbers score higher), assigns a billing plan based on the subscriber ID, and calculates the rated amount using the plan's rate card.

Property Value
Script Engine Groovy
Script Body (see below)
Groovy — Enrichment Script Body 
import groovy.json.JsonSlurper
import groovy.json.JsonOutput
import org.apache.nifi.processor.io.StreamCallback
import java.nio.charset.StandardCharsets

def flowFile = session.get()
if (!flowFile) return

flowFile = session.write(flowFile, { inputStream, outputStream ->
    def reader = new BufferedReader(
      new InputStreamReader(inputStream, StandardCharsets.UTF_8))
    def content = reader.text
    def slurper = new JsonSlurper()
    def parsed = slurper.parseText(content)

    // Handle both single record and array of records
    // (ConsumeKafkaRecord may output arrays)
    def records = parsed instanceof List ? parsed : [parsed]
    def random = new Random()

    def plans = [
      [name:'Prepaid Basic',
       perMinRate:0.12, perSmsRate:0.05, perMbRate:0.02],
      [name:'Prepaid Plus',
       perMinRate:0.08, perSmsRate:0.03, perMbRate:0.015],
      [name:'Postpaid Standard',
       perMinRate:0.05, perSmsRate:0.02, perMbRate:0.01],
      [name:'Postpaid Pro',
       perMinRate:0.03, perSmsRate:0.01, perMbRate:0.005],
      [name:'Postpaid Enterprise',
       perMinRate:0.02, perSmsRate:0.005, perMbRate:0.003]
    ]

    def enriched = records.collect { cdr ->
        def baseScore = random.nextFloat()
        def durationSec = cdr.duration_sec instanceof Number ?
                          cdr.duration_sec.intValue() : 0

        // Boost score for short calls (potential robocalls)
        if (durationSec < 10) {
            baseScore = Math.min(1.0f, baseScore + 0.3f)
        }

        // Boost for premium-rate prefixes
        def calledNumber = (cdr.called_number ?: '').toString()
        if (calledNumber.startsWith('+1900') ||
            calledNumber.startsWith('+506900')) {
            baseScore = Math.min(1.0f, baseScore + 0.4f)
        }

        cdr.fraud_score = Math.round(baseScore * 1000) / 1000.0f
        cdr.fraud_flag = cdr.fraud_score > 0.85

        // Assign plan (consistent per subscriber via hash)
        def subHash = Math.abs(
          (cdr.subscriber_id ?: 'unknown').toString().hashCode())
        def plan = plans[subHash % plans.size()]
        cdr.plan_name = plan.name

        // Calculate rated amount by call type
        def callType = (cdr.call_type ?: 'voice').toString()
        switch (callType) {
            case 'voice':
                def minutes = Math.ceil(durationSec / 60.0)
                cdr.rated_amount = Math.round(
                  minutes * plan.perMinRate * 100) / 100.0
                break
            case 'sms':
                cdr.rated_amount = plan.perSmsRate
                break
            case 'data':
                def mbEstimate = durationSec * 0.5
                cdr.rated_amount = Math.round(
                  mbEstimate * plan.perMbRate * 100) / 100.0
                break
            default:
                cdr.rated_amount = 0.0
        }

        cdr.enriched = true
        return cdr
    }

    def output = parsed instanceof List ? enriched : enriched[0]
    outputStream.write(
      JsonOutput.toJson(output).getBytes(StandardCharsets.UTF_8))
} as StreamCallback)

flowFile = session.putAttribute(flowFile, 'mime.type',
                                'application/json')
session.transfer(flowFile, REL_SUCCESS)

Production pattern: Replace this script with an InvokeHTTP call to a model-serving endpoint (ONNX, TensorFlow Serving, Spark MLflow), or route records to a Spark Structured Streaming job that scores and writes to Kudu directly.

Query the Data in Impala

With data flowing into Kudu, open Hue and run queries that demonstrate both access patterns: subscriber lookups and analytical aggregations.

5a. Row Count and Basic Validation

-- How many CDRs have landed?
SELECT COUNT(*) AS total_cdrs FROM cdr_records;

-- Sample recent records
SELECT subscriber_id, call_start_time, call_type,
       duration_sec, fraud_score, enriched
FROM cdr_records
ORDER BY call_start_time DESC
LIMIT 10;

5b. Subscriber Lookup (Point Query)

First, find a subscriber that has data (with only a few thousand records spread across 10,000 possible IDs, many IDs will be empty): 

-- Find subscribers with the most records
SELECT subscriber_id, COUNT(*) AS calls
FROM cdr_records
GROUP BY subscriber_id
ORDER BY calls DESC
LIMIT 10;

Then use one of those IDs for the point lookup:

-- Subscriber self-service: recent calls for a specific subscriber
SELECT call_start_time, call_type, called_number,
       duration_sec, rated_amount
FROM cdr_records
WHERE subscriber_id = '<ID-FROM-ABOVE>'
ORDER BY call_start_time DESC
LIMIT 10;

What to observe: Check the query profile in Hue. The query should show KuduScanner with partition pruning — only 1 hash bucket scanned. This is the power of hash partitioning on subscriber_id.

5c. Billing Aggregation (Analytical Scan)

-- Total revenue by plan tier
SELECT plan_name,
       COUNT(*) AS total_calls,
       SUM(duration_sec) AS total_seconds,
       ROUND(CAST(SUM(rated_amount) AS DOUBLE), 2) AS total_revenue,
       ROUND(CAST(AVG(rated_amount) AS DOUBLE), 4) AS avg_per_call
FROM cdr_records
WHERE enriched = TRUE
GROUP BY plan_name
ORDER BY total_revenue DESC;

This is a full columnar scan reading only 4 columns out of 13. On Kudu, only those columns are read from disk. On HBase, every column in every row would be deserialized.

5d. Fraud Detection Query

-- Flagged calls in the last hour
SELECT subscriber_id, call_start_time, called_number,
       duration_sec, fraud_score
FROM cdr_records
WHERE fraud_flag = TRUE
ORDER BY fraud_score DESC
LIMIT 50;

5e. Network Quality Analytics

-- Call volume and avg duration by originating cell tower
SELECT orig_cell_id,
       COUNT(*) AS call_count,
       ROUND(AVG(duration_sec), 1) AS avg_duration,
       COUNT(CASE WHEN call_type = 'data' THEN 1 END) AS data_sessions
FROM cdr_records
GROUP BY orig_cell_id
ORDER BY call_count DESC
LIMIT 20;

✓ Checkpoint

  • All queries return results within 1–2 seconds
  • Subscriber lookup scans only 1 hash bucket (check query profile)
  • Billing aggregation reads only projected columns
  • Fraud query returns recently-inserted flagged records with no lag

Partition Lifecycle Management

CDR retention is regulated. The hot tier (Kudu) only needs to hold recent months. Older data should age off — either dropped or archived to a cold tier. This step demonstrates adding and dropping range partitions.

6a. Add Future Partitions

-- Add partitions for upcoming months (do this proactively)
ALTER TABLE cdr_records ADD RANGE PARTITION
  '2026-07-01' <= VALUES < '2026-08-01';
ALTER TABLE cdr_records ADD RANGE PARTITION
  '2026-08-01' <= VALUES < '2026-09-01';

-- Verify partitions
SHOW RANGE PARTITIONS cdr_records;

6b. Drop Expired Partitions

-- Drop an old partition — this is INSTANT
-- No background file deletion, no compaction
ALTER TABLE cdr_records DROP RANGE PARTITION
  '2026-04-01' <= VALUES < '2026-05-01';

Production automation: Schedule a monthly job (cron, Oozie, or Airflow) that: (1) optionally archives the oldest partition to HDFS/Iceberg via Spark (see Lab 03 for the tiered architecture pattern), (2) drops the Kudu partition, (3) adds a new future partition.

✓ Checkpoint

  • New range partitions added successfully
  • Dropped partition reclaims space instantly
  • SHOW RANGE PARTITIONS reflects the changes

Troubleshooting

Error Cause Fix
OutOfMemoryError: Direct buffer memory on ConsumeKafkaRecord Kafka SSL connections exhaust JVM direct buffers Add the dynamic buffer properties from Step 4a. Reduce topic partitions. Verify -XX:MaxDirectMemorySize=1g is set in CM.
TransactionalIdAuthorizationException on PublishKafkaRecord Missing Ranger policy for Kafka Transactional IDs Add a Ranger Kafka policy: Resource Type = Transactional ID, Resource = *, User = nifi, Permissions = Publish, Describe
GroupAuthorizationException: Not authorized to access group Missing Ranger policy for Kafka Consumer Groups Add a Ranger Kafka policy: Resource Type = Consumer Group, Resource = cdr-kudu-writer (or *), User = nifi, Permissions = Consume, Describe
IllegalTypeConversionException: Convert Field Name [call_start_time] to Timestamp Long parsing failed JsonTreeReader does not know the timestamp format Set Timestamp Format to yyyy-MM-dd HH:mm:ss on the JsonTreeReader controller service
NonRecoverableException: Unauthorized action on PutKudu NiFi principal lacks Kudu table permissions Add a Ranger Kudu policy granting nifi all permissions on the target table
Cannot compare java.util.ArrayList with value [...] and java.lang.Integer Enrichment script received a JSON array instead of a single record Use the enrichment script from Step 4b which handles both arrays and single records
Could not login: the client is being asked for a password Keytab not found or wrong keytab path Verify the keytab path is from the NiFi node (not another node). See Lab 00 Step 5.
Subscriber lookup query returns no results The subscriber ID does not exist in the dataset With 10,000 possible IDs and a small dataset, most IDs are empty. Use the GROUP BY query from Step 5b to find IDs that have data.
