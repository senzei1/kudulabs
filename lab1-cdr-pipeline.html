Real-Time Telco CDR Pipeline

Build an end-to-end pipeline that ingests Call Detail Records in near-real-time, enriches them with fraud scores, and serves both subscriber lookups and billing aggregations from a single Kudu table — no Lambda architecture required.

Create the Kudu Table via Impala

Open Hue or impala-shell and create the CDR table. This schema uses hash partitioning on subscriber_id for write distribution and range partitioning on call_start_time for time-based queries and data lifecycle.

Impala SQL 
CREATE TABLE cdr_records (
  subscriber_id   STRING,
  call_start_time TIMESTAMP,
  call_end_time   TIMESTAMP,
  duration_sec    INT,
  call_type       STRING     COMMENT 'voice | sms | data',
  called_number   STRING,
  orig_cell_id    STRING,
  term_cell_id    STRING,
  rated_amount    DECIMAL(12,4),
  plan_name       STRING,
  fraud_score     FLOAT,
  fraud_flag      BOOLEAN,
  enriched        BOOLEAN    DEFAULT FALSE,
  PRIMARY KEY (subscriber_id, call_start_time)
)
PARTITION BY
  HASH (subscriber_id) PARTITIONS 16,
  RANGE (call_start_time) (
    PARTITION '2026-04-01' <= VALUES < '2026-05-01',
    PARTITION '2026-05-01' <= VALUES < '2026-06-01',
    PARTITION '2026-06-01' <= VALUES < '2026-07-01'
  )
STORED AS KUDU
TBLPROPERTIES (
  'kudu.master_addresses' = 'KUDU_MASTER_HOST:7051',
  'kudu.num_tablet_replicas' = '3'
);

Why 16 hash buckets? Target at least 1 tablet per CPU core across your tablet servers. With 3 tablet servers × 8 cores each = 24 cores, 16 hash buckets × 3 range partitions = 48 tablets — about 2 per core, which is a good starting point. Adjust based on your cluster size.

Verify the table was created:

Impala SQL 
DESCRIBE FORMATTED cdr_records;
SHOW TABLE STATS cdr_records;

✓ Checkpoint

You should see the table with 48 tablets (16 hash × 3 range partitions), replication factor 3.

Create the Kafka Topic

Create a topic for the CDR stream. The partition count should match or exceed the NiFi consumer concurrency you plan to use.

Shell 
kafka-topics --create \
  --bootstrap-server KAFKA_BROKER:9092 \
  --topic cdr-raw \
  --partitions 8 \
  --replication-factor 3 \
  --config retention.ms=86400000   # 24h retention

# Verify
kafka-topics --describe \
  --bootstrap-server KAFKA_BROKER:9092 \
  --topic cdr-raw

Build the NiFi Flow — CDR Generator

In a real deployment, CDRs arrive from network mediation platforms. For this lab, we simulate CDR generation using a GenerateFlowFile processor that produces JSON records.

3a. GenerateFlowFile — CDR Simulator

Add a GenerateFlowFile processor with these properties:

PropertyValue
Run Schedule1 sec
Custom Text(see JSON template below)
Batch Size50
Custom Text (JSON CDR template) 
{
  "subscriber_id": "SUB-${random():mod(10000):format('%05d')}",
  "call_start_time": "${now():toNumber():format('yyyy-MM-dd HH:mm:ss')}",
  "call_end_time": "${now():toNumber():plus(${random():mod(600)}000):format('yyyy-MM-dd HH:mm:ss')}",
  "duration_sec": ${random():mod(600)},
  "call_type": "${random():mod(3):equals(0):ifElse('voice', ${random():mod(3):equals(1):ifElse('sms','data')})}",
  "called_number": "+506${random():mod(90000000):plus(10000000)}",
  "orig_cell_id": "CELL-${random():mod(500):format('%04d')}",
  "term_cell_id": "CELL-${random():mod(500):format('%04d')}",
  "rated_amount": null,
  "plan_name": null,
  "fraud_score": null,
  "fraud_flag": null,
  "enriched": false
}

NiFi Expression Language note: The template above is simplified. In practice, you'd use a more sophisticated generator — for example, an ExecuteScript processor with a Groovy/Python script that produces realistic CDR distributions (peak hours, regional cell concentration, call duration distributions). The key point is the schema, not the generator complexity.

4b. UpdateRecord — Simulate Fraud Scoring

In production, fraud scoring would call an external model (Spark ML, ONNX, etc.). For this lab, we simulate it with an UpdateRecord processor that assigns a random score and flags records above a threshold.

PropertyValue
Record ReaderJsonTreeReader
Record WriterJsonRecordSetWriter
/fraud_score${random():mod(100):divide(100)}
/fraud_flag${field.value:replaceEmpty('0'):gt(0.85)}
/rated_amount${field.value:replaceEmpty(${random():mod(5000):divide(100)})}

Production pattern: Replace the UpdateRecord fraud scoring with an InvokeHTTP call to a model-serving endpoint, or route records to a Spark Structured Streaming job that scores and writes to Kudu directly.

4c. PutKudu — Write to Kudu

Connect the UpdateRecord output to a PutKudu processor:

PropertyValue
Kudu MastersKUDU_MASTER:7051
Table Nameimpala::default.cdr_records
Record ReaderJsonTreeReader
Insert OperationUPSERT
Flush ModeAUTO_FLUSH_BACKGROUND
Batch Size500

Table name prefix: When Kudu tables are created through Impala, the internal Kudu table name is prefixed with impala::<database>.. So a table created as cdr_records in the default database is referenced as impala::default.cdr_records in PutKudu.

Route the failure relationship to a LogAttribute processor for debugging. Route success to an auto-terminated connection or a monitoring funnel.

Query the Data in Impala

With data flowing into Kudu, open Hue or impala-shell and run queries that demonstrate both access patterns: subscriber lookups and analytical aggregations.

5a. Row Count and Basic Validation

Impala SQL 
-- How many CDRs have landed?
SELECT COUNT(*) AS total_cdrs FROM cdr_records;

-- Sample recent records
SELECT subscriber_id, call_start_time, call_type,
       duration_sec, fraud_score, enriched
FROM cdr_records
ORDER BY call_start_time DESC
LIMIT 10;

5b. Subscriber Lookup (Point Query)

This simulates the subscriber self-service portal — "show me my recent calls." The query prunes to a single hash bucket (subscriber_id) and a time range.

Impala SQL 
-- Subscriber self-service: last 10 calls for a specific subscriber
SELECT call_start_time, call_type, called_number,
       duration_sec, rated_amount
FROM cdr_records
WHERE subscriber_id = 'SUB-00042'
  AND call_start_time >= '2026-05-01'
ORDER BY call_start_time DESC
LIMIT 10;

What to observe: Check the query profile in Hue (or via PROFILE in impala-shell). The query should show KuduScanner with partition pruning — only 1 hash bucket scanned, only 1 range partition scanned. This is the power of hash+range partitioning.

5c. Billing Aggregation (Analytical Scan)

Impala SQL 
-- Billing: total revenue by plan tier this month
SELECT plan_name,
       COUNT(*) AS total_calls,
       SUM(duration_sec) AS total_seconds,
       SUM(rated_amount) AS total_revenue,
       AVG(rated_amount) AS avg_revenue_per_call
FROM cdr_records
WHERE call_start_time >= '2026-05-01'
  AND call_start_time < '2026-06-01'
  AND enriched = TRUE
GROUP BY plan_name
ORDER BY total_revenue DESC;

This is a full columnar scan across one range partition, reading only 4 columns out of 14. On Kudu, only those 4 columns are read from disk. On HBase, the entire row (all 14 columns) would be deserialized for every record.

5d. Fraud Detection Query

Impala SQL 
-- Real-time fraud monitoring: flagged calls in the last hour
SELECT subscriber_id, call_start_time, called_number,
       duration_sec, fraud_score
FROM cdr_records
WHERE fraud_flag = TRUE
  AND call_start_time >= now() - INTERVAL 1 HOUR
ORDER BY fraud_score DESC
LIMIT 50;

5e. Network Quality Analytics

Impala SQL 
-- Call volume and avg duration by originating cell tower
-- Useful for network capacity planning
SELECT orig_cell_id,
       COUNT(*) AS call_count,
       AVG(duration_sec) AS avg_duration,
       COUNT(CASE WHEN call_type = 'data' THEN 1 END) AS data_sessions
FROM cdr_records
WHERE call_start_time >= '2026-05-01'
GROUP BY orig_cell_id
ORDER BY call_count DESC
LIMIT 20;

✓ Checkpoint

  • All queries return results within 1–2 seconds
  • Subscriber lookup scans only 1 hash bucket (check query profile)
  • Billing aggregation reads only projected columns (check bytes read in profile)
  • Fraud query returns recently-inserted flagged records with no lag

Partition Lifecycle Management

CDR retention is regulated. In most LATAM jurisdictions, CDRs must be available for 1–5 years, but the hot tier (Kudu) only needs to hold recent months. Older data should age off to HDFS/Iceberg for cold storage.

6a. Add Future Partitions

Impala SQL 
-- Add partitions for upcoming months (do this proactively)
ALTER TABLE cdr_records ADD RANGE PARTITION
  '2026-07-01' <= VALUES < '2026-08-01';
ALTER TABLE cdr_records ADD RANGE PARTITION
  '2026-08-01' <= VALUES < '2026-09-01';

-- Verify partitions
SHOW RANGE PARTITIONS cdr_records;

6b. Drop Expired Partitions

Impala SQL 
-- Once data has been archived to HDFS/Iceberg, drop the Kudu partition
-- This is INSTANT — no background file deletion, no compaction
ALTER TABLE cdr_records DROP RANGE PARTITION
  '2026-04-01' <= VALUES < '2026-05-01';

Production automation: Schedule a monthly job (cron, Oozie, or Airflow) that: (1) exports the oldest Kudu range partition to Iceberg/Parquet via Spark, (2) validates the export, (3) drops the Kudu partition, (4) adds a new future partition.

6c. Archive to Iceberg (Cold Tier)

PySpark 
# Age data from Kudu to Iceberg for long-term retention
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("CDR Kudu-to-Iceberg Aging") \
    .config("spark.sql.catalog.iceberg_catalog", "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.iceberg_catalog.type", "hive") \
    .getOrCreate()

# Read the partition to archive from Kudu
archive_df = spark.read.format("kudu") \
    .option("kudu.master", "KUDU_MASTER:7051") \
    .option("kudu.table", "impala::default.cdr_records") \
    .load() \
    .filter("call_start_time >= '2026-04-01' AND call_start_time < '2026-05-01'")

# Write to Iceberg (append — this is cold storage, immutable)
archive_df.writeTo("iceberg_catalog.default.cdr_archive") \
    .option("write-format", "parquet") \
    .append()

print(f"Archived {archive_df.count()} CDRs to Iceberg")

# After validation, drop the Kudu partition via Impala
# ALTER TABLE cdr_records DROP RANGE PARTITION '2026-04-01' <= VALUES < '2026-05-01';

✓ Checkpoint

  • New range partitions added successfully
  • Archived data is readable from the Iceberg table
  • Dropped Kudu partition reclaims space instantly
  • Impala can query both Kudu (hot) and Iceberg (cold) tables

Cluster Health Validation

After running the pipeline for 10+ minutes, validate cluster health and table statistics using the Kudu CLI.

Shell 
# Run ksck — the primary health diagnostic
kudu cluster ksck KUDU_MASTER:7051

# Check table statistics
kudu table statistics KUDU_MASTER:7051 impala::default.cdr_records

# Inspect tablet distribution across tablet servers
kudu table list KUDU_MASTER:7051 --columns=name,num_tablets

# Check for any under-replicated tablets
kudu cluster ksck KUDU_MASTER:7051 2>&1 | grep -i "under-replicated"
