Lab 00

Security Setup for Secured CDP Clusters

Every production CDP cluster uses Kerberos authentication and TLS encryption. This page walks through the one-time setup required before running any lab. Complete these steps first — all labs reference this configuration.

Duration: 15 minutes
Level: Foundational
Applies to: All labs

This is not optional. If your CDP cluster has Kerberos enabled (it almost certainly does), you must complete these steps before any lab will work. Commands that skip the security configuration will fail with disconnection errors, OOM errors (SSL handshake mismatch), or authentication rejections.

Gather Your Cluster Details

Before starting, collect these values from Cloudera Manager. You'll use them throughout every lab.

Value Where to Find It Example
Kerberos principal Your AD/FreeIPA account user@EXAMPLE.REALM.COM
Kudu master address(es) CM → Kudu → Instances → Master node1.example.com:7051
Kafka broker address(es) CM → Kafka → Instances → Broker node2.example.com:9093
Kafka listener port CM → Kafka → Configuration → listeners 9093 (SASL_SSL)
Impala daemon address CM → Impala → Instances → Daemon node3.example.com:21050
NiFi URL CM → NiFi → Web UI link https://node4.example.com:8443/nifi

Tip: Write these down or export them as environment variables in your shell session. The labs use placeholders like <YOUR-KUDU-MASTER> — replace them with your actual values.

Shell — export your cluster addresses (optional convenience) 
# Set these once per session — adjust to your cluster
export KUDU_MASTERS="node1.example.com:7051"
export KAFKA_BROKERS="node2.example.com:9093"
export IMPALA_HOST="node3.example.com"

Kerberos Authentication

Every CLI tool (impala-shell, kafka-topics, kudu, spark-submit) requires a valid Kerberos ticket. Obtain one before starting any lab.

Shell 
# Authenticate with your Kerberos principal
kinit your_username@YOUR.REALM.COM

# Verify the ticket
klist

You should see output like:

Ticket cache: FILE:/tmp/krb5cc_1000
Default principal: user@EXAMPLE.REALM.COM

Valid starting       Expires              Service principal
05/05/2026 22:30:00  05/06/2026 22:30:00  krbtgt/EXAMPLE.REALM.COM@EXAMPLE.REALM.COM

Ticket expiration: Kerberos tickets typically expire after 10–24 hours. If a lab command suddenly starts failing with authentication errors after working earlier, run kinit again.

TLS Truststore Configuration

CDP auto-TLS generates a cluster-wide truststore on every node. You need its path and password for any client that connects over SSL (Kafka, Kudu, etc.).

3a. Locate the Truststore

Shell 
# The standard CDP auto-TLS truststore location:
ls -l /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks

3b. Find the Truststore Password

The password varies by cluster. Try these approaches in order:

Shell 
# Option 1: Check for a password file next to the truststore
ls -la /var/lib/cloudera-scm-agent/agent-cert/
cat /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_password 2>/dev/null

# Option 2: Check the CM agent config
grep -r "truststore_password" /etc/cloudera-scm-agent/config.ini 2>/dev/null

# Option 3: Check the Kafka broker config (if Kafka is on this node)
grep -i "ssl.truststore.password" \
  /run/cloudera-scm-agent/process/$(ls -t /run/cloudera-scm-agent/process/ | grep kafka-KAFKA_BROKER | head -1)/kafka.properties 2>/dev/null

# Option 4: Try common defaults
# Test each with keytool until one works:
keytool -list -keystore /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks -storepass "cloudera" 2>/dev/null && echo "Password: cloudera"
keytool -list -keystore /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks -storepass "changeit" 2>/dev/null && echo "Password: changeit"
keytool -list -keystore /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks -storepass "" 2>/dev/null && echo "Password: (empty)"

Note: If the broker config shows a ssl.truststore.password.generator script instead of a plain password, the password is dynamically generated by CM. Try the common defaults above — cloudera works in many auto-TLS setups.

3c. Export as Variables

Shell 
# Once you've found the working password:
export TRUSTSTORE_PATH="/var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks"
export TRUSTSTORE_PASS="cloudera"  # Replace with your actual password

✓ Checkpoint

Verify the truststore is readable with your password:

Shell 
keytool -list -keystore $TRUSTSTORE_PATH -storepass "$TRUSTSTORE_PASS" | head -5
# Should show: Keystore type, provider, and at least one entry

Kafka Client Configuration

Every Kafka CLI command in a secured cluster requires a client properties file and a JAAS configuration. Create both once, use them everywhere.

4a. Create the JAAS Configuration

Shell 
cat > /tmp/kafka-jaas.conf << 'EOF'
KafkaClient {
  com.sun.security.auth.module.Krb5LoginModule required
  useKeyTab=false
  useTicketCache=true;
};
EOF

4b. Create the Client Properties File

Shell 
cat > /tmp/kafka-client.properties << EOF
security.protocol=SASL_SSL
sasl.mechanism=GSSAPI
sasl.kerberos.service.name=kafka
ssl.truststore.location=${TRUSTSTORE_PATH}
ssl.truststore.password=${TRUSTSTORE_PASS}
ssl.truststore.type=jks
EOF

4c. Set Environment Variables

Shell 
# Required for every kafka-* command
export KAFKA_HEAP_OPTS="-Xmx512m"
export KAFKA_OPTS="-Djava.security.auth.login.config=/tmp/kafka-jaas.conf"

Make it permanent: Add the exports to your ~/.bashrc so they persist across sessions. You can also alias the Kafka commands to always include --command-config:

Shell — add to ~/.bashrc 
# CDP Kafka CLI configuration
export KAFKA_HEAP_OPTS="-Xmx512m"
export KAFKA_OPTS="-Djava.security.auth.login.config=/tmp/kafka-jaas.conf"
export KAFKA_CMD_CONFIG="--command-config /tmp/kafka-client.properties"

# Convenience aliases
alias kafka-topics-sec="kafka-topics --command-config /tmp/kafka-client.properties"
alias kafka-console-consumer-sec="kafka-console-consumer --consumer.config /tmp/kafka-client.properties"
alias kafka-console-producer-sec="kafka-console-producer --producer.config /tmp/kafka-client.properties"

✓ Checkpoint

Test the full configuration by listing Kafka topics:

Shell 
kafka-topics --list \
  --bootstrap-server <YOUR-KAFKA-BROKER>:9093 \
  --command-config /tmp/kafka-client.properties

If this returns a topic list (or empty output with no errors), your Kafka security config is working. If you see OOM or disconnection errors, revisit the truststore password or verify your Kerberos ticket is valid.

Impala Shell Configuration

In a kerberized cluster, impala-shell needs the -k flag to use Kerberos authentication.

Shell 
# Connect to Impala with Kerberos auth
impala-shell -i <YOUR-IMPALA-HOST>:21050 -k

# Or via Hue (recommended for the labs):
# Hue handles Kerberos authentication automatically through the browser.
# Access via: https://<HUE-HOST>:8889

Hue is easier: For the SQL portions of these labs, Hue is the recommended interface. It handles authentication transparently and provides a query editor with syntax highlighting, query profiles, and result visualization. Use impala-shell -k when you need CLI access.

Ranger Permissions

Your user needs appropriate permissions in Ranger for each service used in the labs. Work with your Ranger admin (or do it yourself if you have admin access) to ensure these policies exist:

Service Resource Permissions Needed
Kudu Tables: * (or specific table names) Admin, Create, Select, Insert, Update, Delete
Kafka Topics: cdr-raw, * for lab flexibility Create, Configure, Publish, Consume, Describe
Kafka Consumer Groups: * Consume, Describe
Impala (Hive/SQL) Database: default, Tables: * Create, Select, Insert, Update, Delete, All
HDFS Home directory + write paths Read, Write, Execute
NiFi Canvas access View, Modify (via NiFi policies in NiFi UI or Ranger)

Lab 04 (Compliance) requires Ranger admin access to create column-level security policies. If you don't have Ranger admin, ask your admin to set up the policies described in Lab 04 Step 4 before you start.

Quick Validation — All Services

Run these checks to confirm all services are reachable before starting any lab:

Shell 
# 1. Kerberos ticket valid
klist | head -5

# 2. Kudu cluster healthy
kudu cluster ksck <YOUR-KUDU-MASTER>:7051 2>&1 | tail -5

# 3. Kafka reachable
kafka-topics --list \
  --bootstrap-server <YOUR-KAFKA-BROKER>:9093 \
  --command-config /tmp/kafka-client.properties

# 4. Impala responsive
impala-shell -i <YOUR-IMPALA-HOST>:21050 -k -q "SELECT 1"

# 5. NiFi accessible (from your browser)
# Navigate to: https://<YOUR-NIFI-HOST>:8443/nifi

✓ Setup Complete

  • Kerberos ticket obtained and valid
  • Truststore located and password confirmed
  • Kafka client properties and JAAS config created
  • Kafka, Kudu, Impala, and NiFi all reachable
  • Ranger permissions in place for your user

You're ready. Proceed to Lab 01: CDR Pipeline.
