Lab 00

Security Setup for Secured CDP Clusters

Every production CDP cluster uses Kerberos authentication and TLS encryption. This page walks through the one-time setup required before running any lab. Complete these steps first — all labs reference this configuration.

Duration: 15–30 minutes
Level: Foundational
Applies to: All labs

This is not optional. If your CDP cluster has Kerberos enabled (it almost certainly does), you must complete these steps before any lab will work. Commands that skip the security configuration will fail with disconnection errors, OOM errors (SSL handshake mismatch), or authentication rejections.

Gather Your Cluster Details

Before starting, collect these values from Cloudera Manager. You will use them throughout every lab. All labs use angle-bracket placeholders like <YOUR-KUDU-MASTER> — replace them with your actual values.

Value Where to Find It Example
Kerberos realm CM → Administration → Settings → Kerberos EXAMPLE.COM
Kerberos principal Your AD/FreeIPA account myuser@EXAMPLE.COM
Kudu master address(es) CM → Kudu → Instances → Master master1.cluster.example.com:7051
Kafka broker address(es) CM → Kafka → Instances → Broker broker1.cluster.example.com:9093
Kafka listener port CM → Kafka → Configuration → listeners 9093 (SASL_SSL) — not 9092
Impala daemon address CM → Impala → Instances → Daemon impalad1.cluster.example.com:21050
NiFi URL CM → NiFi → Web UI link https://nifi1.cluster.example.com:8443/nifi
Shell — export your cluster addresses (optional convenience) 
# Set these once per session — adjust to your cluster
export KUDU_MASTERS="master1.cluster.example.com:7051"
export KAFKA_BROKERS="broker1.cluster.example.com:9093"
export IMPALA_HOST="impalad1.cluster.example.com"

Kerberos Authentication

Every CLI tool (impala-shell, kafka-topics, kudu, spark-submit) requires a valid Kerberos ticket.

Shell 
kinit your_username@YOUR.REALM
klist

Ticket expiration: Kerberos tickets typically expire after 10–24 hours. If a lab command suddenly starts failing with authentication errors after working earlier, run kinit again.

TLS Truststore Configuration

CDP auto-TLS generates a cluster-wide truststore on every node. You need its path and password for any client that connects over SSL.

3a. Locate the Truststore

ls -l /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks

3b. Find the Truststore Password

The password varies by cluster. Try these approaches in order: 

# Option 1: Check for a password file next to the truststore
cat /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_password 2>/dev/null

# Option 2: Check the CM agent config
grep -r "truststore_password" /etc/cloudera-scm-agent/config.ini 2>/dev/null

# Option 3: Extract from the Kafka broker config (if Kafka runs on this node)
grep -i "ssl.truststore.password" \
  /run/cloudera-scm-agent/process/$(ls -t /run/cloudera-scm-agent/process/ \
  | grep kafka-KAFKA_BROKER | head -1)/kafka.properties 2>/dev/null

# Option 4: Try common defaults
keytool -list -keystore /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks \
  -storepass "cloudera" 2>/dev/null && echo "Password: cloudera"
keytool -list -keystore /var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks \
  -storepass "changeit" 2>/dev/null && echo "Password: changeit"

Note: If the Kafka broker config shows a ssl.truststore.password.generator script instead of a plain password, skip the script and try the common defaults above.

3c. Export as Variables

export TRUSTSTORE_PATH="/var/lib/cloudera-scm-agent/agent-cert/cm-auto-global_truststore.jks"
export TRUSTSTORE_PASS="your-truststore-password"

✓ Checkpoint

Verify:

keytool -list -keystore $TRUSTSTORE_PATH -storepass "$TRUSTSTORE_PASS" | head -5

Kafka Client Configuration

Every Kafka CLI command requires two files: a JAAS configuration and a client properties file. Without these, you get OOM errors or disconnection loops.

4a. Create the JAAS Configuration

cat > /tmp/kafka-jaas.conf << 'EOF'
KafkaClient {
  com.sun.security.auth.module.Krb5LoginModule required
  useKeyTab=false
  useTicketCache=true;
};
EOF

4b. Create the Client Properties File

cat > /tmp/kafka-client.properties << EOF
security.protocol=SASL_SSL
sasl.mechanism=GSSAPI
sasl.kerberos.service.name=kafka
ssl.truststore.location=${TRUSTSTORE_PATH}
ssl.truststore.password=${TRUSTSTORE_PASS}
ssl.truststore.type=jks
EOF

4c. Set Environment Variables

export KAFKA_HEAP_OPTS="-Xmx512m"
export KAFKA_OPTS="-Djava.security.auth.login.config=/tmp/kafka-jaas.conf"

4d. Make It Permanent (Optional)

# Add to ~/.bashrc
export KAFKA_HEAP_OPTS="-Xmx512m"
export KAFKA_OPTS="-Djava.security.auth.login.config=/tmp/kafka-jaas.conf"

# Convenience aliases
alias kafka-topics-sec="kafka-topics --command-config /tmp/kafka-client.properties"
alias kafka-console-consumer-sec="kafka-console-consumer --consumer.config /tmp/kafka-client.properties"
alias kafka-console-producer-sec="kafka-console-producer --producer.config /tmp/kafka-client.properties"

Every kafka-* CLI command requires --command-config /tmp/kafka-client.properties. Without it, the CLI defaults to plaintext on port 9092 and fails. This applies to every secured CDP cluster, not just yours.

✓ Checkpoint

kafka-topics --list \
  --bootstrap-server <YOUR-KAFKA-BROKER>:9093 \
  --command-config /tmp/kafka-client.properties

NiFi Kerberos Configuration

NiFi processors cannot use the Kerberos ticket cache. They need a keytab file. CDP creates one for the NiFi service on each node, but the path differs per node.

Keytab paths differ per node. Each CDP node stores its keytab in a unique process directory. The numeric process ID is different on every node. You must use the keytab from the node where NiFi is actually running.

5a. Identify Which Node Runs NiFi

Check CM → NiFi → Instances. SSH into that host for the next steps.

5b. Find the Keytab on the NiFi Node

# SSH into the NiFi node, then:
NIFI_KEYTAB=$(ls /run/cloudera-scm-agent/process/$(ls -t /run/cloudera-scm-agent/process/ \
  | grep nifi-NIFI | head -1)/*.keytab)
echo "Keytab path: $NIFI_KEYTAB"

# List the principals inside it
klist -kt "$NIFI_KEYTAB"

5c. Verify the Keytab Works

kinit -kt "$NIFI_KEYTAB" nifi/$(hostname -f)@YOUR.REALM
klist

5d. Values for NiFi Processors

Use these values in every NiFi Kafka and Kudu processor:

Processor Property Value Notes
Kerberos Principal nifi/_HOST@<YOUR-REALM> _HOST auto-resolves to the local FQDN at runtime
Kerberos Keytab /run/cloudera-scm-agent/process/<PROCESS-ID>-nifi-NIFI_NODE/nifi.keytab Use the actual path from step 5b — process ID is unique per node
Kerberos Service Name kafka This is the broker's service name, not NiFi's
SSL Context Service Default Truststore-only NiFi SSL Context Service Pre-created by CDP auto-TLS — no need to create a new one

Impala Shell Configuration

6a. Fix Python Version

If impala-shell fails with "This impala-shell package was not built to support Python 2.7": 

export IMPALA_PYTHON_EXECUTABLE=python3
# Make permanent:
echo 'export IMPALA_PYTHON_EXECUTABLE=python3' >> ~/.bashrc

6b. Connect with Kerberos + SSL + HS2

impala-shell -i <YOUR-IMPALA-HOST>:21050 -k --ssl --protocol=hs2

If you see "Invalid method name: PingImpalaService", add --protocol=hs2. This means the server uses the HiveServer2 protocol but the shell defaulted to the legacy beeswax protocol.

Recommendation: Use Hue for the SQL portions of these labs. It handles authentication, SSL, and protocol selection transparently.

Ranger Permissions

7a. Your User Permissions

Service Resource Permissions
Kudu Tables: * Admin, Create, Select, Insert, Update, Delete
Kafka Topics: * Create, Configure, Publish, Consume, Describe
Kafka Consumer Groups: * Consume, Describe
Impala (Hive/SQL) Database: default, Tables: * Create, Select, Insert, Update, Delete, All
HDFS Home directory + write paths Read, Write, Execute

7b. NiFi Service Principal Permissions

NiFi authenticates to Kafka using its own service keytab. The nifi principal (short name after auth-to-local mapping) needs its own Kafka policies:

Service Resource Type Resource User Permissions
Kafka Topic * nifi Publish, Consume, Create, Describe
Kafka Consumer Group * nifi Consume, Describe
Kafka Transactional ID * nifi Publish, Describe

Do not skip the Transactional ID policy. PublishKafkaRecord uses Kafka transactions by default. Without this policy, you will get TransactionalIdAuthorizationException. Topic permissions alone are not sufficient.

Quick Validation — All Services

# 1. Kerberos ticket valid
klist | head -5

# 2. Kudu cluster healthy
kudu cluster ksck <YOUR-KUDU-MASTER>:7051 2>&1 | tail -5

# 3. Kafka reachable
kafka-topics --list \
  --bootstrap-server <YOUR-KAFKA-BROKER>:9093 \
  --command-config /tmp/kafka-client.properties

# 4. Impala responsive
IMPALA_PYTHON_EXECUTABLE=python3 impala-shell \
  -i <YOUR-IMPALA-HOST>:21050 -k --ssl --protocol=hs2 \
  -q "SELECT 1"

# https://<YOUR-NIFI-HOST>:8443/nifi
