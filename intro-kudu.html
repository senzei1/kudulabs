What is Apache Kudu?

A fast, columnar storage engine built for the Hadoop ecosystem that handles both real-time random access and efficient analytical scans — in a single system.

The One-Liner

Kudu fills the gap that existed between HDFS (great for batch analytics, terrible for random access) and HBase (great for random access, terrible for analytics). It gives you both in one place, with a relational data model and native SQL through Impala.

Architecture Overview

A Kudu cluster has two types of processes: Masters that manage metadata, and Tablet Servers that store and serve data. Tables are split into tablets — horizontal partitions distributed across tablet servers. Each tablet is replicated for fault tolerance using the Raft consensus protocol.

Masters

Manage the table catalog, schema definitions, and tablet locations. Raft-replicated (typically 3) for high availability. One leader handles writes; followers are hot standbys.

Tablet Servers

Store and serve tablet replicas. Each tablet server can host tablets from many tables. For a given tablet, one server is the Raft leader (handles writes); others serve reads and provide fault tolerance.

Tablets

Horizontal partitions of a table, defined by the partitioning scheme (hash, range, or both). Each tablet is independently replicated. Target 1–10 GB per tablet for optimal performance.

Raft

Consensus protocol for both master HA and tablet replication. With replication factor 2f+1, the system tolerates f failures. Kudu uses logical replication — operations, not data blocks, travel over the network.

How Data is Stored

Each tablet uses a hybrid row/columnar storage format. New writes land in an in-memory row store, then flush to disk as columnar files. Updates and deletes are tracked in separate delta stores and merged during background compaction.

Base Data

Columnar CFiles — one file per column, compressed and encoded. This is where analytical scan performance comes from: read only the columns you need, skip the rest.

UNDO Deltas

Record how to reverse recent mutations. Used for MVCC — reading at a past timestamp. Allow consistent snapshots without blocking writes.

REDO Deltas

Record mutations that haven't been merged into base data yet. When you UPDATE a row, the change goes here first. Compaction eventually applies REDOs into the base data.

Compaction

Background process that merges DiskRowSets and applies delta stores into base data. Unlike HBase, Kudu's compaction is columnar and incremental — lighter on I/O and doesn't block queries.

Partitioning: How Tables Become Tablets

Kudu supports two partitioning strategies that can be combined. The partitioning scheme determines how rows are distributed across tablets — and getting it right is the single most important design decision for Kudu performance.

Where Kudu Shines

✓ Strengths

  • Fast random access and fast analytical scans on the same table
  • Native row-level INSERT, UPDATE, DELETE, UPSERT — no file rewrites
  • Data is queryable immediately after write (milliseconds, not minutes)
  • Columnar storage with encoding and compression
  • Tight Impala integration — predicate pushdown, column projection, parallel scans
  • Partition lifecycle — add/drop range partitions instantly
  • Built-in Raft replication — no dependency on HDFS
  • Immutable columns (1.17+) for audit/compliance use cases
  • Simpler ops than HBase — no ZooKeeper, no compaction storms

⚠ Limitations

  • Not designed for multi-petabyte cold storage — use HDFS/Iceberg for that
  • Not an OLTP database — no multi-table transactions or complex joins at the storage level
  • Requires a predefined schema — no schema-on-read or sparse columns
  • Primary key cannot be changed after table creation
  • Hash bucket count is fixed at table creation — can't resize without recreating
  • Manages its own replication — doesn't benefit from HDFS erasure coding
  • Requires Kudu-aware clients (Impala, Spark, NiFi) — not an open file format like Iceberg/Parquet
  • No time-travel / snapshot queries (unlike Iceberg)

Best Use Cases

The pattern is always the same: data that changes frequently and must be queried analytically with low latency. If your data is append-only and queried in batch, HDFS + Parquet is cheaper. If your data is pure key-value with no analytics, HBase is faster. Kudu is for everything in between.

Use Case Why Kudu Not This
CDC from OLTP databases Native UPSERT, immediate queryability Iceberg (file-rewrite on every update)
Real-time dashboards on streaming data Sub-second ingest + sub-second Impala queries Hive ACID (minutes of latency)
IoT / telemetry / CDR processing Hash+range partitioning, fast time-range scans HBase (poor analytical scans)
Feature stores for ML models Read features + write scores back to same table Separate serving + training stores
Slowly changing dimensions Row-level UPDATE without rewriting files Iceberg COW (rewrites entire files)
Compliance / audit with immutable fields Immutable columns + mutable workflow fields in one table Separate audit log + operational table

Ecosystem Integration in CDP

The Pitch:

When you're talking to a customer who has streaming or frequently-updated data and needs both operational access and analytical queries, the pitch is simple:

"You don't need a Lambda architecture." Kudu gives you fast ingest, row-level mutations, and analytical queries in a single system. No separate serving layer. No nightly ETL. No stale dashboards. Write once, query immediately, through the same SQL interface your team already knows.

The strongest positioning is the tiered architecture: Kudu for the hot/warm tier (recent data, frequent updates, interactive queries), Iceberg or Parquet on HDFS for the cold tier (historical data, batch analytics, cheap storage). Impala queries both transparently. This is not "replace everything with Kudu" — it's "use each tool where it's strongest."
