Lab 04

Compliance Audit Platform

Build a regulatory compliance system using Kudu's immutable columns for tamper-evident audit fields, Ranger for column-level access control, and Impala for compliance search queries. Demonstrates patterns applicable to LGPD (Brazil), CRC (Colombia), and IFT (Mexico) requirements.

Duration: 90 minutes
Level: Intermediate
Components: NiFi, Kudu, Ranger, Impala
Filing Ingest (NiFi)
→
Kudu (immutable + mutable)
→
Ranger (column security)
→
Impala (compliance queries)

Prerequisites

  • CDP cluster with NiFi, Kudu 1.17.0+ (required for immutable columns), Impala, and Ranger
  • Ranger admin access for creating access policies
  • At least two test users: one "compliance_officer" and one "analyst" with different access levels
  • Kudu Java client or Impala for table creation (Impala doesn't yet expose immutable column syntax — use the Kudu API or CLI)

Important caveat: Kudu's immutable columns prevent modification through the client API. They do not provide cryptographic WORM (Write Once Read Many) guarantees at the storage layer. An administrator with direct filesystem access could theoretically modify data files. For regulatory environments requiring true cryptographic immutability, additional infrastructure controls (WORM storage, blockchain-anchored checksums) are needed. Be transparent about this boundary with customers.

Create the Compliance Table with Immutable Columns

Immutable columns are a Kudu 1.17+ feature. Once a value is written to an immutable column, it cannot be updated — any attempt to UPDATE that column will be rejected by the tablet server. This is configured at table creation time using the Kudu client API.

Impala limitation: As of CDP 7.1.9, Impala's CREATE TABLE syntax doesn't expose the immutable column attribute. You need to create the table using the Kudu Java client, Python client, or the kudu CLI, then create an external Impala table pointing to it.

1a. Create Table via Kudu Java Client

Java — CreateComplianceTable.java 
import org.apache.kudu.ColumnSchema;
import org.apache.kudu.Schema;
import org.apache.kudu.Type;
import org.apache.kudu.client.*;

import java.util.*;

public class CreateComplianceTable {
    public static void main(String[] args) throws Exception {
        String masters = "KUDU_MASTER:7051";

        try (KuduClient client = new KuduClient.KuduClientBuilder(masters).build()) {

            List<ColumnSchema> columns = new ArrayList<>();

            // Primary key
            columns.add(new ColumnSchema.ColumnSchemaBuilder("record_id", Type.STRING)
                    .key(true).build());

            // ── IMMUTABLE columns ── (tamper-evident audit fields)
            columns.add(new ColumnSchema.ColumnSchemaBuilder("filing_date", Type.UNIXTIME_MICROS)
                    .nullable(false)
                    .immutable(true)  // Cannot be changed after insert
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("filing_type", Type.STRING)
                    .nullable(false)
                    .immutable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("jurisdiction", Type.STRING)
                    .nullable(false)
                    .immutable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("entity_name", Type.STRING)
                    .nullable(false)
                    .immutable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("entity_tax_id", Type.STRING)
                    .nullable(false)
                    .immutable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("document_hash", Type.STRING)
                    .nullable(false)
                    .immutable(true)  // SHA-256 of original filing — tamper detection
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("submitted_by", Type.STRING)
                    .nullable(false)
                    .immutable(true)
                    .build());

            // ── MUTABLE columns ── (workflow fields that change)
            columns.add(new ColumnSchema.ColumnSchemaBuilder("review_status", Type.STRING)
                    .nullable(true)
                    .defaultValue("pending")
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("assigned_reviewer", Type.STRING)
                    .nullable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("review_notes", Type.STRING)
                    .nullable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("risk_score", Type.FLOAT)
                    .nullable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("compliance_outcome", Type.STRING)
                    .nullable(true)
                    .build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder("last_updated", Type.UNIXTIME_MICROS)
                    .nullable(true)
                    .build());

            Schema schema = new Schema(columns);

            CreateTableOptions options = new CreateTableOptions();
            options.addHashPartitions(List.of("record_id"), 8);
            options.setNumReplicas(3);

            client.createTable("compliance_records", schema, options);
            System.out.println("Table 'compliance_records' created with immutable audit columns");
        }
    }
}

2c. PutKudu — Insert Filings

PropertyValue
Kudu MastersKUDU_MASTER:7051
Table Namecompliance_records
Insert OperationINSERT (not UPSERT — first write only)
Record ReaderJsonTreeReader

INSERT, not UPSERT: For the initial filing, use INSERT to ensure the immutable columns are written exactly once. Subsequent updates to workflow fields (review_status, assigned_reviewer, etc.) should use UPSERT — the immutable columns will be preserved because Kudu rejects any attempt to change them.

Demonstrate Immutable Column Protection

This is the critical demonstration: prove that audit fields cannot be altered after the initial insert.

3a. Pick a Record

Impala SQL 
-- Find a specific record to test with
SELECT record_id, entity_name, document_hash, review_status
FROM compliance_records
LIMIT 1;
-- Note the record_id for the next steps

3b. Attempt to Modify an Immutable Column (This Should Fail)

Impala SQL 
-- Try to change the entity name (immutable column)
-- This will be REJECTED by Kudu
UPDATE compliance_records
SET entity_name = 'TAMPERED-NAME'
WHERE record_id = 'FIL-xxxx-xxxx';  -- use your record_id

Expected result: Kudu returns an error: Immutable: UPDATE not allowed for immutable column: entity_name. The update is rejected entirely — the row is unchanged. This is enforced at the tablet server level, not the client.

3c. Update a Mutable Column (This Should Succeed)

Impala SQL 
-- Update the review status (mutable column) — this works fine
UPDATE compliance_records
SET review_status = 'in_review',
    assigned_reviewer = 'reviewer-005',
    last_updated = now()
WHERE record_id = 'FIL-xxxx-xxxx';  -- use your record_id

-- Verify: mutable fields changed, immutable fields preserved
SELECT record_id, entity_name, document_hash,
       review_status, assigned_reviewer, last_updated
FROM compliance_records
WHERE record_id = 'FIL-xxxx-xxxx';

Key takeaway: The same table has both immutable audit fields (entity_name, document_hash, filing_date, submitted_by) and mutable workflow fields (review_status, assigned_reviewer, notes). No need for separate audit tables or append-only logging — Kudu enforces the immutability at the storage layer.

✓ Checkpoint

  • UPDATE on immutable column was rejected with an error
  • UPDATE on mutable columns succeeded
  • Immutable column values remain unchanged after the mutable update

Configure Ranger Column-Level Security

Apply fine-grained access control so that different roles see different columns. This is critical for LGPD/CRC compliance — PII fields like entity_tax_id should only be visible to authorized compliance officers.

4a. Create Ranger Policies

In the Ranger admin UI, navigate to the Kudu service and create these policies:

Policy NameTableColumnsUsers/GroupsPermissions
Compliance Officer - Full Access compliance_records * (all) compliance_officers group Select, Insert, Update
Analyst - No PII compliance_records entity_tax_id analysts group Deny
Analyst - Read Only compliance_records * (all except denied) analysts group Select (read only)

Ranger column masking: Instead of denying access entirely, you can use Ranger's column masking policies to show partial data — e.g., mask entity_tax_id as ***-***-1234 (show last 4 digits). This lets analysts see that a tax ID exists without revealing the full value.

4b. Verify Access Control

Impala SQL (as compliance_officer)

FROM compliance_records
LIMIT 5;
Impala SQL (as analyst)

Compliance Search & Reporting Queries

These queries demonstrate the analytical capability that Kudu provides on top of the compliance data — something that's hard to achieve with append-only log systems or relational databases at scale.

5a. Filing Volume by Jurisdiction

Impala SQL 
SELECT jurisdiction,
       filing_type,
       COUNT(*) AS total_filings,
       COUNT(CASE WHEN review_status = 'pending' THEN 1 END) AS pending,
       COUNT(CASE WHEN review_status = 'in_review' THEN 1 END) AS in_review,
       COUNT(CASE WHEN compliance_outcome = 'approved' THEN 1 END) AS approved,
       COUNT(CASE WHEN compliance_outcome = 'rejected' THEN 1 END) AS rejected
FROM compliance_records
GROUP BY jurisdiction, filing_type
ORDER BY jurisdiction, total_filings DESC;

5b. Reviewer Workload Distribution

Impala SQL 
SELECT assigned_reviewer,
       COUNT(*) AS assigned_cases,
       COUNT(CASE WHEN review_status = 'pending' THEN 1 END) AS backlog,
       AVG(risk_score) AS avg_risk_score
FROM compliance_records
WHERE assigned_reviewer IS NOT NULL
GROUP BY assigned_reviewer
ORDER BY backlog DESC;

5c. High-Risk Filings Requiring Attention

Impala SQL 
SELECT record_id, entity_name, jurisdiction, filing_type,
       risk_score, review_status, assigned_reviewer
FROM compliance_records
WHERE risk_score > 0.75
  AND review_status IN ('pending', 'in_review')
ORDER BY risk_score DESC
LIMIT 25;

5d. Simulate Complete Review Workflow

Impala SQL 
-- Step 1: Assign a reviewer
UPDATE compliance_records
SET review_status = 'in_review',
    assigned_reviewer = 'reviewer-003',
    last_updated = now()
WHERE record_id = 'FIL-xxxx-xxxx';

-- Step 2: Add review notes and risk score
UPDATE compliance_records
SET review_notes = 'Documentation complete. Low risk entity. Approved.',
    risk_score = 0.15,
    last_updated = now()
WHERE record_id = 'FIL-xxxx-xxxx';

-- Step 3: Record compliance outcome
UPDATE compliance_records
SET review_status = 'completed',
    compliance_outcome = 'approved',
    last_updated = now()
WHERE record_id = 'FIL-xxxx-xxxx';

-- Verify full audit trail
SELECT record_id,
       -- Immutable fields (unchanged since creation)
       filing_date, entity_name, document_hash, submitted_by,
       -- Mutable fields (updated through workflow)
       review_status, assigned_reviewer, risk_score,
       compliance_outcome, review_notes, last_updated
FROM compliance_records
WHERE record_id = 'FIL-xxxx-xxxx';

Audit integrity: The immutable fields (filing_date, entity_name, document_hash, submitted_by) remain exactly as they were at filing time. The mutable fields reflect the review workflow. A regulator can verify that the original filing data was never tampered with by comparing document_hash against the original document.

Retention & Data Lifecycle

For production systems, add range partitioning by filing_date to enable efficient data lifecycle management:

Design Pattern 
-- If creating a new table with range partitioning:
-- PRIMARY KEY (record_id, filing_date)
-- PARTITION BY
--   HASH (record_id) PARTITIONS 8,
--   RANGE (filing_date) (
--     PARTITION '2025-01-01' <= VALUES < '2026-01-01',  -- Year 2025
--     PARTITION '2026-01-01' <= VALUES < '2027-01-01',  -- Year 2026
--   )
--
-- Retention enforcement:
-- After legal retention period expires (e.g., 5 years for Brazil LGPD):
--   1. Archive the range partition to Iceberg
--   2. ALTER TABLE DROP RANGE PARTITION (instant deletion)
--   3. Log the deletion for regulatory audit

LGPD right to deletion: If a subject requests data deletion under LGPD, you can DELETE the specific rows from the Kudu table. The immutable columns don't prevent row deletion — they only prevent column-level modification. Document this distinction for compliance officers.
