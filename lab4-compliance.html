Compliance Audit Platform

Build a regulatory compliance system using Kudu's immutable columns for tamper-evident audit fields, Ranger for column-level access control, and Impala for compliance search queries. Demonstrates patterns applicable to LGPD (Brazil), CRC (Colombia), and IFT (Mexico) requirements.

Important caveat: Kudu's immutable columns prevent modification through the client API. They do not provide cryptographic WORM (Write Once Read Many) guarantees at the storage layer. An administrator with direct filesystem access could theoretically alter data files. For regulatory environments requiring true cryptographic immutability, additional infrastructure controls are needed. Be transparent about this boundary with customers.

Create the Compliance Table with Immutable Columns

What are immutable columns? A Kudu 1.17+ feature. Once a value is written to an immutable column, it cannot be updated — any attempt to UPDATE that column is rejected by the tablet server with an error. This is configured at table creation time.

Impala limitation: As of CDP 7.1.9, Impala's CREATE TABLE syntax does not expose the immutable column attribute. You must create the table using the Kudu Java client, then create an external Impala table pointing to it.

1a. Create Table via Kudu Java Client

Save this as CreateComplianceTable.java, compile, and run it on a node with Kudu client libraries available.

Java — CreateComplianceTable.java 
import org.apache.kudu.ColumnSchema;
import org.apache.kudu.Schema;
import org.apache.kudu.Type;
import org.apache.kudu.client.*;

import java.util.*;

public class CreateComplianceTable {
    public static void main(String[] args) throws Exception {
        // Replace with your Kudu master address
        String masters = "<YOUR-KUDU-MASTER>:7051";

        try (KuduClient client =
               new KuduClient.KuduClientBuilder(masters).build()) {

            List<ColumnSchema> columns = new ArrayList<>();

            // Primary key
            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "record_id", Type.STRING)
                    .key(true).build());

            // ── IMMUTABLE columns ──
            // These cannot be changed after insert
            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "filing_date", Type.UNIXTIME_MICROS)
                    .nullable(false).immutable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "filing_type", Type.STRING)
                    .nullable(false).immutable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "jurisdiction", Type.STRING)
                    .nullable(false).immutable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "entity_name", Type.STRING)
                    .nullable(false).immutable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "entity_tax_id", Type.STRING)
                    .nullable(false).immutable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "document_hash", Type.STRING)
                    .nullable(false).immutable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "submitted_by", Type.STRING)
                    .nullable(false).immutable(true).build());

            // ── MUTABLE columns ──
            // These change as the review progresses
            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "review_status", Type.STRING)
                    .nullable(true).defaultValue("pending").build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "assigned_reviewer", Type.STRING)
                    .nullable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "review_notes", Type.STRING)
                    .nullable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "risk_score", Type.FLOAT)
                    .nullable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "compliance_outcome", Type.STRING)
                    .nullable(true).build());

            columns.add(new ColumnSchema.ColumnSchemaBuilder(
                    "last_updated", Type.UNIXTIME_MICROS)
                    .nullable(true).build());

            Schema schema = new Schema(columns);

            CreateTableOptions options = new CreateTableOptions();
            options.addHashPartitions(List.of("record_id"), 8);
            options.setNumReplicas(3);

            client.createTable("compliance_records", schema, options);
            System.out.println(
              "Table created with immutable audit columns");
        }
    }
}

Ingest Compliance Filings via NiFi

Build a NiFi flow that simulates regulatory filings arriving from an intake portal. Each filing gets a SHA-256 hash stored in the immutable document_hash column for tamper detection.

2a. ExecuteScript — Filing Generator

Property Value
Script Engine Groovy
Run Schedule 1 sec
Script Body (see below)
Groovy — Filing Generator Script Body 
import groovy.json.JsonOutput
import org.apache.nifi.processor.io.StreamCallback
import java.nio.charset.StandardCharsets
import java.security.MessageDigest

def flowFile = session.create()
def random = new Random()
def fmt = "yyyy-MM-dd HH:mm:ss"
def now = new Date().format(fmt)

def types = ['tax_return', 'suspicious_activity',
             'annual_report', 'regulatory_filing']
def jurisdictions = ['BR', 'CO', 'MX', 'CR', 'PE']
def officers = (1..20).collect {
    String.format("officer-%03d", it) }

def recordId = "FIL-" + UUID.randomUUID().toString()
def entityName = "Entity-" + String.format("%04d",
                  random.nextInt(1000))
def taxId = String.valueOf(
             random.nextInt(900000000) + 1000000000L)

// Generate SHA-256 hash of the filing content
def content = "${recordId}|${entityName}|${taxId}|${now}"
def digest = MessageDigest.getInstance("SHA-256")
def hashBytes = digest.digest(
                  content.getBytes(StandardCharsets.UTF_8))
def docHash = hashBytes.collect {
                String.format('%02x', it) }.join()

def filing = [
  record_id         : recordId,
  filing_date       : now,
  filing_type       : types[random.nextInt(types.size())],
  jurisdiction      : jurisdictions[
                        random.nextInt(jurisdictions.size())],
  entity_name       : entityName,
  entity_tax_id     : taxId,
  document_hash     : docHash,
  submitted_by      : officers[random.nextInt(officers.size())],
  review_status     : "pending",
  assigned_reviewer : null,
  review_notes      : null,
  risk_score        : null,
  compliance_outcome: null,
  last_updated      : null
]

flowFile = session.write(flowFile, { inputStream, outputStream ->
    outputStream.write(
      JsonOutput.toJson(filing).getBytes(StandardCharsets.UTF_8))
} as StreamCallback)

flowFile = session.putAttribute(flowFile, 'mime.type',
                                'application/json')
session.transfer(flowFile, REL_SUCCESS)

INSERT, not UPSERT: For the initial filing, use INSERT to ensure immutable columns are written exactly once. Subsequent updates to workflow fields (review_status, assigned_reviewer) use UPSERT via Impala — the immutable columns are preserved because Kudu rejects any attempt to change them.

Table name: Because this table was created via the Kudu Java client (not Impala), its internal name is compliance_records — without the impala::default. prefix. Use the bare name in PutKudu.

Connect ExecuteScript → PutKudu. Route failure to LogAttribute; auto-terminate success. Let the flow run for 1–2 minutes to accumulate filings.

Demonstrate Immutable Column Protection

This is the critical demonstration: prove that audit fields cannot be altered after the initial insert.

3a. Pick a Record

SELECT record_id, entity_name, document_hash, review_status
FROM compliance_records
LIMIT 1;
-- Note the record_id for the next steps

3b. Attempt to Modify an Immutable Column (Should Fail)

-- Try to change the entity name (immutable column)
-- This will be REJECTED by Kudu
UPDATE compliance_records
SET entity_name = 'TAMPERED-NAME'
WHERE record_id = '<RECORD-ID-FROM-ABOVE>';

Expected result: Kudu returns an error: Immutable: UPDATE not allowed for immutable column: entity_name. The update is rejected entirely — the row is unchanged. This is enforced at the tablet server level, not the client.

3c. Update a Mutable Column (Should Succeed)

-- Update the review status (mutable column) — this works
UPDATE compliance_records
SET review_status = 'in_review',
    assigned_reviewer = 'reviewer-005',
    last_updated = now()
WHERE record_id = '<RECORD-ID-FROM-ABOVE>';

-- Verify: mutable fields changed, immutable fields preserved
SELECT record_id, entity_name, document_hash,
       review_status, assigned_reviewer, last_updated
FROM compliance_records
WHERE record_id = '<RECORD-ID-FROM-ABOVE>';

Key takeaway: The same table has both immutable audit fields (entity_name, document_hash, filing_date, submitted_by) and mutable workflow fields (review_status, assigned_reviewer, notes). No need for separate audit tables — Kudu enforces immutability at the storage layer.

✓ Checkpoint

  • UPDATE on immutable column was rejected with an error
  • UPDATE on mutable columns succeeded
  • Immutable column values remain unchanged

Configure Ranger Column-Level Security

Apply fine-grained access control so different roles see different columns. This is critical for LGPD/CRC compliance — PII fields like entity_tax_id should only be visible to authorized compliance officers.

4a. Create Ranger Policies

In Ranger admin UI, navigate to the Kudu service:

Policy Name Table Columns Users/Groups Permissions
Compliance Officer - Full Access compliance_records * (all) Your user / compliance_officers group Select, Insert, Update, Delete
Analyst - Deny PII compliance_records entity_tax_id analysts group (or a test user) Deny
Analyst - Read Only compliance_records * analysts group (or a test user) Select (read only)

Ranger column masking: Instead of denying access entirely, Ranger can mask column values — e.g., show entity_tax_id as ***-***-1234 (last 4 digits). Configure this under the Masking tab when creating the policy.

4b. Verify Access Control

If you have two test users, connect to Impala as each and run the same query: 

-- As compliance officer (full access):
SELECT record_id, entity_name, entity_tax_id,
       filing_type, jurisdiction
FROM compliance_records
LIMIT 5;

-- As analyst (restricted):
-- entity_tax_id should be denied or masked
SELECT record_id, entity_name, entity_tax_id,
       filing_type, jurisdiction
FROM compliance_records
LIMIT 5;

✓ Checkpoint

  • Compliance officer sees all columns including entity_tax_id
  • Analyst is denied or sees masked values for entity_tax_id
  • Analyst cannot INSERT or UPDATE (read-only)

Compliance Search & Reporting Queries

Run each query separately in Hue.

5a. Filing Volume by Jurisdiction

SELECT jurisdiction, filing_type,
       COUNT(*) AS total_filings,
       COUNT(CASE WHEN review_status = 'pending' THEN 1 END) AS pending,
       COUNT(CASE WHEN review_status = 'in_review' THEN 1 END) AS in_review,
       COUNT(CASE WHEN compliance_outcome = 'approved' THEN 1 END) AS approved,
       COUNT(CASE WHEN compliance_outcome = 'rejected' THEN 1 END) AS rejected
FROM compliance_records
GROUP BY jurisdiction, filing_type
ORDER BY jurisdiction, total_filings DESC;

5b. Reviewer Workload

SELECT assigned_reviewer,
       COUNT(*) AS assigned_cases,
       COUNT(CASE WHEN review_status = 'pending' THEN 1 END) AS backlog,
       ROUND(AVG(risk_score), 2) AS avg_risk
FROM compliance_records
WHERE assigned_reviewer IS NOT NULL
GROUP BY assigned_reviewer
ORDER BY backlog DESC;

5c. High-Risk Filings

SELECT record_id, entity_name, jurisdiction, filing_type,
       risk_score, review_status, assigned_reviewer
FROM compliance_records
WHERE risk_score > 0.75
  AND review_status IN ('pending', 'in_review')
ORDER BY risk_score DESC
LIMIT 25;

5d. Complete Review Workflow

Walk through the full lifecycle of a filing — assign, review, approve. Run each statement separately. 

-- Step 1: Assign a reviewer
UPDATE compliance_records
SET review_status = 'in_review',
    assigned_reviewer = 'reviewer-003',
    last_updated = now()
WHERE record_id = '<RECORD-ID>';

-- Step 2: Add review notes and risk score
UPDATE compliance_records
SET review_notes = 'Documentation complete. Low risk. Approved.',
    risk_score = 0.15,
    last_updated = now()
WHERE record_id = '<RECORD-ID>';

-- Step 3: Record compliance outcome
UPDATE compliance_records
SET review_status = 'completed',
    compliance_outcome = 'approved',
    last_updated = now()
WHERE record_id = '<RECORD-ID>';

Verify the full audit trail:

SELECT record_id,
       -- Immutable fields (unchanged since creation)
       filing_date, entity_name, document_hash, submitted_by,
       -- Mutable fields (updated through workflow)
       review_status, assigned_reviewer, risk_score,
       compliance_outcome, review_notes, last_updated
FROM compliance_records
WHERE record_id = '<RECORD-ID>';

Audit integrity: The immutable fields remain exactly as they were at filing time. The mutable fields reflect the review workflow. A regulator can verify that the original filing data was never tampered with by comparing document_hash against the original document.

Data Lifecycle & Retention

For production compliance systems, consider these design patterns:

Range Partitioning for Retention

If you need to enforce retention periods (e.g., 5 years for Brazil LGPD, 10 years for Mexico tax records), add range partitioning by filing_date to the table design. After the retention period expires, archive the partition to Iceberg (see Lab 03) then drop it instantly with ALTER TABLE DROP RANGE PARTITION.

LGPD Right to Deletion

If a data subject requests deletion under LGPD, you can DELETE specific rows from the Kudu table. Immutable columns prevent column-level modification, but they do not prevent row deletion. Document this distinction for compliance officers.
